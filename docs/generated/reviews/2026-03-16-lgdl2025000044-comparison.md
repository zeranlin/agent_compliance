## 五、人工已发现但当前代码仍偏弱或漏掉的点

### 1. 样品评分主观性和分档失衡没有被完整抓出

人工会把样品评分单独列为重要问题，因为当前条款：

- 使用“优、良、中、基本满足、差”这类主观分档
- 采用 `80% / 40% / 20% / 10% / 0` 的跳档
- 缺少面料、尺寸、做工、缺陷等客观量化锚点

当前代码只抓到了履约评价类的 `ambiguous_requirement`，但没有把样品评分单独拎出来。

### 2. 认证评分结构问题尚未被准确识别

人工会将“环境标志产品认证 + 四类体系认证 + 成立不足三个月按已获得计分”视为一组结构性评分问题。  
当前代码尚未把这类问题稳定识别出来，尤其是：

- 认证类评分权重本身偏高
- 允许成立不足三个月的供应商以证明替代真实取得证书

### 3. 技术偏离评分中“同一份检测报告呈现”这一形式门槛尚未单独成点

当前代码能抓到大量 `technical_justification_needed`，但主要聚焦于具体指标本身。  
人工审查会额外指出：

- “必须在同一份报告中呈现”属于证明形式门槛
- 对医用服装和床品供货项目而言，该门槛未必具有充分必要性

### 4. 明显跨领域模板错贴仍有漏报

当前代码只稳定抓到了保洁条款，对“芯片及系统”“以采购人实际需求为准”这类跨领域错贴和开放式需求变更，尚未形成稳定 finding。

当前代码对验收和需求边界的表达还不够完整，更偏向单句规则触发。

## 六、差异原因分析

### 1. 当前代码更擅长抓“显性异常”，对“评分结构性失衡”仍偏弱

代码已经能抓本地业绩、科技型中小企业、营业执照等显性加分，但对“样品评分主观分档”和“认证评分结构失衡”这类需要理解整张评分表设计逻辑的问题，仍缺少更强的结构分析能力。

### 2. 技术类规则已能广覆盖，但对“证明形式门槛”的拆分还不够

现在代码对安全环保、抗菌、防护类指标抓得较全，所以会形成较多 `technical_justification_needed`。  
但人工除了看指标，还会拆出：

- 指标本身是否过密
- 报告时点是否过严
- 报告组织形式是否不合理

代码目前主要覆盖了第一层，对第二、第三层还不够稳定。

### 3. 模板残留识别仍偏词典式，跨领域错贴还不够强

保洁条款之所以能抓到，是因为它属于高频模板残留词。  
但“芯片及系统无缝对接”这种错贴，需要代码具备更强的“标的域识别 + 义务域识别”能力，也就是要知道：

- 当前项目主标的是医用服装、床品等纺织类货物
- 芯片和系统对接属于另一类信息化或物联网义务

这一层当前代码还没有完全建立。

### 4. 商务链路问题需要多条款联合推理

人工会把责任条款、验收条款、开放式需求条款一起看，判断供应商承担了哪些不确定风险。  
当前代码仍以单条规则命中为主，虽然已经能抓到 `负全责` 这类绝对责任，但还不够擅长把相关条款联成一个更完整的商务风险链。
