这与人工审查的判断已经非常接近。

## 四、人工已发现但代码仍偏弱或仍有偏差的部分

### 1. “特种设备安全管理和作业人员证书”仍被定成了资格错位，而非评分错位

当前代码上浮了：

- `资格条件中存在与标的域不匹配的资质或登记要求`

但这条原文实际出现在评分项，不在资格准入章节。  
人工会更自然地判断成：

- 团队或评分项中混入明显不匹配证书

也就是说，代码已经看见了问题，但问题类型和章节属性还有偏差。

### 2. 模板错贴主题和评分错位主题仍有重叠

当前代码同时上浮了：

- `评分项中存在与标的域不匹配的证书认证或模板内容`
- `文件中存在与标的域不匹配的模板残留或义务外扩`

对于这份文件来说，二者都不是错，但边界还没有完全拉开。  
人工更容易区分：

- 哪些是评分表里的错位证书
- 哪些是文件模板里的错贴内容

当前代码还存在“一段证据同时支撑两条主问题”的情况。

### 3. 验收边界主题仍然有少量评分材料串入

当前代码已经能上浮：

- `验收程序、复检与最终确认边界不清`

但它的证据段仍包含部分评分项里的“验收报告或履约评价”取证文字。  
人工通常会把：

- 评分取证要求
- 履约验收规则

拆开处理，不会把它们混成同一个验收边界主题。

### 4. 技术“需论证”类问题仍偏噪

当前代码在这份文件里还保留了几条：

- `安全环保类技术要求可能合理但需补充必要性论证`

但它们主要来自评分表或投标文件格式中的“质量（完成时间、安全、环保）保障措施及方案”标题。  
人工不会把这一类标题性文字单独上浮成技术论证问题，说明这部分仍然存在模板说明污染。

## 五、为什么会出现这些差异

### 1. 代码现在已经具备章节级归并能力，但问题类型路由还不够稳

这份文件最明显的例子就是：

- `特种设备安全管理和作业人员证书`

代码已经知道它不对，但还可能把它往“资格错位”而不是“评分错位/人员证书错位”上路由。  
这说明问题不再是“没看见”，而是“看见之后分到哪个主题还不够稳”。

### 2. 模板错贴和评分错位还共用了一部分证据池

在信息化项目里，错位证书、错位认证范围和模板残留很容易同时出现。  
当前代码已经能把这些问题上浮为主问题，但 `finding_arbiter` 和 `evidence_selector` 还没有完全把它们拆开成互不污染的两组证据。

### 3. 商务和验收链路已经增强，但评分章节里的“验收类取证要求”仍会干扰主题边界

这份文件里，评分表中存在：

- `验收报告`
- `履约评价`

这会对验收边界主题造成干扰。  
人工会自然区分“评分取证材料”和“合同履约验收规则”，代码当前还在适应这种跨章节同词场景。

### 4. 技术“需论证”过滤还不够克制

当前 `technical_necessity_explainer` 已经能说明“为什么要论证”，但在这份文件上仍会把一些评分标题、模板章节标题错当成技术约束点。  
这说明“技术标题词”与“真实技术要求”还需要更强的区分。

当前剩下的主要差距不再是“召回不足”，而是：

- 问题类型路由还不够准
