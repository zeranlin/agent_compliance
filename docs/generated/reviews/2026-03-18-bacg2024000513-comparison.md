## 四、人工已发现但代码仍偏弱或出现误判的部分

### 1. 代码把资格问题误判得过重

人工复核后认为：

- 招标公告里的资格要求基本还是法定准入条款
- 真正的问题不在资格门槛

但代码上浮出了：

- `资格条件整体超出法定准入和履约必需范围`

造成误判的主要原因是：

- 投标文件模板里的中小企业声明填写说明、社保材料、原件备查说明等被吸进了资格语义范围
- `finding_arbiter` 在这份文件上没有把“模板说明文本”和“真实资格门槛”分开

### 2. 代码误报了“品牌档次赋分”

人工复核评分表，没有发现“国内一线品牌/国际知名品牌直接打分”的条款。  
但代码上浮了：

- `评分项直接按品牌档次赋分`

这说明当前 `brand_and_certification_scoring_analyzer` 还会把：

- 样品优良中差
- 品牌字段填写说明
- 核心产品品牌填写提示

错误串联成“品牌评分”主问题。

### 3. 代码没有稳定识别“家具项目中混入定位管理标签模块软件著作权/专利评分”这一更关键的错位评分点

人工认为这条是非常突出的主问题：

- 家具采购里引入 `定位管理标签模块`
- 再以软件著作权或专利证书打满分

当前代码虽然抓到了一些评分错位和技术错位，但没有把这条稳定上浮成独立高风险主问题。  
这说明当前代码对“家具 + 信息化模块”的混合评分场景还不够敏感。

### 4. 代码没有把“资产定位系统、智能芯片、碳足迹填报”等附加义务稳定提成主问题

人工会明确认为：

- 安装资产定位系统
- 在所有家具安装智能芯片
- 填写产品碳排放数据

这些要求已经明显超出普通家具供货安装的核心边界，应单独审视。  
当前代码虽然在商务链条里提到了责任偏重，但没有把这组“附加义务外扩”拆成清晰的主题主问题。

### 5. 代码把商务链条归并得过大，导致代表性证据和位置失真

当前代码第一条主问题：

- `履约全链路中的付款、验收、责任和到场响应边界整体偏向供应商承担`

位置跨到了评分、资格、技术、合同、质疑处理多个章节，辅助行号从 `14-8224`。  
这已经明显超出“可改稿主问题”的合理边界。人工虽然也会做总括，但不会把几乎整份文件并进一条结论。

## 五、为什么会出现这些差异

### 1. 当前代码对评分和商务高频模式更敏感，但对“模板说明”和“真实条款”的边界过滤还不够

这份文件里有大量：

- 投标文件模板说明
- 原件备查说明
- 评分说明
- 中小企业声明填写说明

它们会在关键词层面干扰资格、品牌、社保、原件等主题判断。  
人工能自然排除这些“说明性文字”，代码现在还不够稳。

### 2. 代码对“家具 + 信息化模块”的混合采购边界还不够敏感

这份文件不是纯家具采购，它混入了：

- `定位管理标签模块`
- `资产定位管理系统`
- `智能芯片`

人工会立刻把它视为混合采购边界问题。  
代码目前虽然已有 `mixed_scope_boundary_engine`，但还没有把这种“家具主体 + 信息化附加模块”稳定长成评分和商务双重主问题。

### 3. 代码对技术和商务条款会做大主题归并，但仲裁层还不够克制

这次代码已经能形成章节级主问题，但：

- 归并范围偏大
- 代表性证据不够精准
- 跨章节串联太多

所以输出还不够像人工正式审查意见，更像“主题正确但定位过宽”。

### 4. 代码对“设计方案 / 样品 / 创新模块”这类多主题评分项还缺少更细的语义一致性判断

人工能区分：

- 设计方案高主观分
- 样品高主观分
- 创新模块错位
- 设备保障措施不当加分

而当前代码仍倾向于：

- 抓主观分档
- 抓高分值
- 但对“评分名称、评分内容、评分证据、评分目的是否一致”的判断还不够细。

## 六、代码审查下一步需要改进的设计和功能点

### 1. 增强 `brand_and_certification_scoring_analyzer`

重点不是再补“品牌”关键词，而是增强：

- 区分真实品牌评分和品牌字段填写说明
- 区分认证评分与创新模块评分
- 识别家具采购中夹带软件著作权/专利的错位评分

### 2. 增强 `scoring_semantic_consistency_engine`

重点补：

- 设计方案、样品、创新能力、技术保障措施四类评分项的名称-内容-证据-目的是否一致
- 把“生产设备打分”“样品主观打分”“软件著作权打分”拆成更自然的子主题

### 3. 增强 `mixed_scope_boundary_engine`

重点补：

- 家具主体 + 信息化模块
- 家具主体 + 资产定位系统
- 家具主体 + 智能芯片

让系统能更稳定地把“合理配套”和“义务外扩”分开。

### 4. 增强 `commercial_lifecycle_analyzer`

重点补：

- 保修期
- 响应时限
- 附加系统建设义务
- 碳足迹填报
- 检测增补责任

让这类条款不再只是并入“商务责任偏重”，而是能拆成更利于改稿的主题包。

### 5. 继续增强 `finding_arbiter`

重点补：

- 排除模板说明型文本对资格、品牌和评分主题的污染
- 控制跨章节过度归并
- 让代表性证据更短、更准、更像人工会引用的原文

## 七、当前判断

这份家具项目上，代码审查已经能看见：

- 主观评分
- 技术证明形式过严
- 医院场景高指标叠加
- 商务验收责任偏重

但它还没有完全达到人工审查的成熟度。当前最大的差距不是“完全没找到问题”，而是：

- 资格主问题误判
- 品牌评分误报
- 创新模块错位评分没有稳定上浮
- 商务主问题归并过宽

所以这份文件对后续增强最有价值的信号是：

- 要强化评分语义一致性
- 要强化混合采购边界识别
- 要强化模板说明排除
- 要让仲裁层更克制、更贴近改稿视角
