## 四、人工已发现但代码仍偏弱或出现误判的部分

### 1. 资格误判已经明显改善，但评分模板说明仍会污染个别主问题

当前版本里，人工复核后认为：

- 招标公告里的资格要求基本还是法定准入条款
- 真正的问题不在资格门槛

这条在当前版本里已经不再被上浮成资格主问题。  
但评分类主问题里，模板说明文本仍会污染代表性摘录，例如：

- `政府采购投标及履约承诺函`
- `格式自定`
- `填写说明`

说明模板说明过滤已起效，但还没有完全下沉到“评分主题的证据选择”层。

### 2. “品牌档次赋分”误报已经修掉，但“评分错位主题”还不够聚焦

人工复核评分表，没有发现“国内一线品牌/国际知名品牌直接打分”的条款。  
当前版本已经不再误报这一条，说明 `brand_and_certification_scoring_analyzer` 对显性品牌评分和说明性文字的区分已明显改善。

但评分类主问题仍不够聚焦，主要表现为：

- `评分项名称、内容和评分证据之间不一致` 仍会吞入过长、过杂的证据段
- `评分项中存在与标的域不匹配的证书认证或模板内容` 还没有把“软件著作权/专利评分”稳定提成最突出的子问题

### 3. 代码没有稳定识别“家具项目中混入定位管理标签模块软件著作权/专利评分”这一更关键的错位评分点

人工认为这条是非常突出的主问题：

- 家具采购里引入 `定位管理标签模块`
- 再以软件著作权或专利证书打满分

当前代码虽然已经新增：

- `家具采购场景叠加资产定位和智能管理系统义务，边界不清`

但还没有把这条“家具评分中以定位管理标签模块的软件著作权/专利证书打满分”的问题稳定提升为评分章节的独立主问题。  
这说明当前代码对“家具 + 信息化模块”的混合评分场景还不够敏感，`mixed_scope_boundary_engine` 和 `scoring_semantic_consistency_engine` 之间的联动还不够。

### 4. 代码没有把“资产定位系统、智能芯片、碳足迹填报”等附加义务稳定提成主问题

人工会明确认为：

- 安装资产定位系统
- 在所有家具安装智能芯片
- 填写产品碳排放数据

这些要求已经明显超出普通家具供货安装的核心边界，应单独审视。  
当前代码已经能上浮成：

- `家具采购场景叠加资产定位和智能管理系统义务，边界不清`
- `文件中存在与标的域不匹配的模板残留或义务外扩`

但这两条仍然存在边界重叠：前者偏混合采购，后者偏模板残留，目前还没有拆得像人工那样清楚。

### 5. 代码把商务链条归并得过大，导致代表性证据和位置失真

当前代码第一条主问题：

- `履约全链路中的付款、验收、责任和到场响应边界整体偏向供应商承担`

这一轮已经从早期跨到 `1403-8035` 的状态，收紧到了以商务章节为主的 `2843-7962`。  
同时，`验收送检、检测和专家评审费用整体转嫁给供应商` 也已从跨资格章节的大范围收束到了验收配合条款本身。

但它仍然会把：

- 商务要求
- 合同条款
- 部分验收说明

一并并进同一个大主题，离人工改稿时常用的“付款/验收/责任/到场响应分别改”还有距离。

## 五、为什么会出现这些差异

### 1. 当前代码对评分和商务高频模式更敏感，但对“模板说明”和“真实条款”的边界过滤还需继续下沉到证据层

这份文件里有大量：

- 投标文件模板说明
- 原件备查说明
- 评分说明
- 中小企业声明填写说明

它们会在关键词层面干扰资格、品牌、社保、原件等主题判断。  
人工能自然排除这些“说明性文字”，代码当前已经能在主问题层规避一部分误判，但代表性摘录和证据选择层还不够稳。

### 2. 代码对“家具 + 信息化模块”的混合采购边界已经有感知，但评分和商务的联动还不够

这份文件不是纯家具采购，它混入了：

- `定位管理标签模块`
- `资产定位管理系统`
- `智能芯片`

人工会立刻把它视为混合采购边界问题。  
代码目前虽然已有 `mixed_scope_boundary_engine`，也已经长出了混合采购边界主问题，但还没有把这种“家具主体 + 信息化附加模块”稳定拆成：

- 评分错位问题
- 义务外扩问题
- 商务边界问题

### 3. 代码对技术和商务条款会做大主题归并，但仲裁层还不够克制

这次代码已经能形成章节级主问题，而且比上一轮收得更准，但：

- 归并范围仍偏大
- 代表性证据还不够精准
- 混合采购主题和模板残留主题仍有重叠

所以输出还不够像人工正式审查意见，更像“主题正确但定位过宽”。

### 4. 代码对“设计方案 / 样品 / 创新模块”这类多主题评分项还缺少更细的语义一致性判断

人工能区分：

- 设计方案高主观分
- 样品高主观分
- 创新模块错位
- 设备保障措施不当加分

而当前代码仍倾向于：

- 抓主观分档
- 抓高分值
- 但对“评分名称、评分内容、评分证据、评分目的是否一致”的判断还不够细。

## 六、代码审查下一步需要改进的设计和功能点

### 1. 继续增强 `scoring_semantic_consistency_engine`

重点补：

- 让 `定位管理标签模块 + 软件著作权/专利评分` 这类家具场景下的创新模块错位，稳定上浮为评分主问题
- 让评分主题、评分证据和代表性摘录更聚焦，减少模板说明文本混入

### 2. 继续增强 `mixed_scope_boundary_engine`

重点补：

- 家具主体 + 信息化模块
- 家具主体 + 资产定位系统
- 家具主体 + 智能芯片

让系统能更稳定地把“合理配套”和“义务外扩”分开，并与评分问题联动。

### 3. 继续增强 `commercial_lifecycle_analyzer`

重点补：

- 保修期
- 响应时限
- 附加系统建设义务
- 检测增补责任

让这类条款不再只是并入“大商务链条主题”，而是能拆成更利于改稿的子主题。

### 4. 继续增强 `finding_arbiter`

重点补：

- 排除模板说明型文本对评分主题和证据摘录的污染
- 控制商务主题跨章节过度归并
- 让代表性证据更短、更准、更像人工会引用的原文

### 5. 保留 `brand_and_certification_scoring_analyzer` 的当前修正方向

这条本轮已经消除了明显品牌误报，后续重点应放在：

- 区分认证评分与创新模块评分
- 识别家具采购中夹带软件著作权/专利的错位评分

## 七、当前判断

这份家具项目上，代码审查已经能看见：

- 主观评分
- 技术证明形式过严
- 医院场景高指标叠加
- 商务验收责任偏重

但它还没有完全达到人工审查的成熟度。当前最大的差距不是“完全没找到问题”，而是：

- 创新模块错位评分没有稳定上浮
- 混合采购边界主题与模板残留主题仍有重叠
- 商务主问题归并仍偏宽
- 代表性证据选择还不够像人工改稿意见

所以这份文件对后续增强最有价值的信号是：

- 要强化评分语义一致性
- 要强化混合采购边界识别
- 要强化模板说明排除
- 要让仲裁层更克制、更贴近改稿视角
