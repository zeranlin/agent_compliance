## 六、为什么当前代码和人工差这么多

### 1. 当前规则覆盖面仍然偏向“已知高频样本”，对本文件的特殊错位条款覆盖不足

目前代码规则更擅长抓：

- 通用资格门槛
- 已在 benchmark 里出现过的评分问题
- 一些常见商务责任条款

但这份柴油发电机项目里出现了几类当前规则尚未稳定覆盖的特殊问题：

- 行业完全不相称的资格证书  
  如 `有害生物防制`、`SPCA登记证书`
- 商务评分项中夹带企业规模和财务指标
- 制造商资质评分项中夹带明显跨领域证书

这些都属于“人工一看就知道不对，但当前规则还没长出来”的问题。

### 2. 当前 LLM 任务不是全文自由审，而是“候选问题增强”，候选没进来就不会补出来

当前代码链路里的本地模型主要承担：

- 评分结构判断
- 模板错贴识别
- 商务链路联合判断

它并不是对整份文件进行完全自由审查，而是建立在“先有候选条款或候选问题”的基础上增强判断。  
这意味着：

- 如果规则阶段没有把 `有害生物防制资质` 这类条款送进候选池，模型也不会稳定补出来。
- 如果评分项解析没有把“商务条款偏离情况”里的企业规模、资产、纳税额识别成异常评分候选，模型也很难自动形成正确 finding。

所以这不是“大模型没用”，而是“候选生成太弱，模型没接住这些问题”。

### 3. 表格解析和区块切分还不够稳，导致命中位置错位

这次代码结果里最明显的两个噪声是：

- 命中到了“政府采购履约异常情况反馈表”的 `优/良/中/差`
- 命中到了“评审程序及评审方法”章节的泛化文本

这说明当前区块切分和评分项识别还存在两个问题：

- 会把非评分表里的“优良中差”误当成评分分档
- 会在长文档后段误触发主观评分规则

也就是说，代码不仅“漏报”，还有一定“错报”。

### 4. 当前代码更擅长“分值异常”，不擅长“内容错位”

人工在这份文件中强的地方，其实不是看出“100 分很高”，而是看出：

- 为什么这里会出现 `有机产品认证`
- 为什么会出现 `水运机电工程专项监理资质`
- 为什么柴油发电机项目会要求 `有害生物防制资质`

这类问题本质上是“条款内容与项目标的不匹配”的语义判断。  
当前代码还没有把“项目主标的域”和“条款域”做强约束，所以对跨领域错贴和错位评分的识别能力明显不足。

### 5. 当前技术与商务 LLM 任务在这份文件上没有形成有效补点

这份文件本来很适合触发：

- 技术要求写死年份
- 制造商工程师限定
- 付款和运行期绑定
- 运输责任绝对免责

但实际没有稳定产出对应 finding，说明当前技术与商务链路的 LLM 任务至少还有两个问题：

- 触发条件过窄
- 提示词对“柴油发电机组供货及安装”这类项目场景适配不足

## 七、这次不一致的结论性判断

一句话说，这次代码和人工差异大的根本原因不是“模型能力不够”，而是：

`规则候选覆盖不足 + 表格区块识别不稳 + 标的域/条款域错位识别能力不足 + 技术商务链路触发不充分`

更具体地说：

- 人工已经能做“项目语境下的整体判断”
- 代码目前仍主要停留在“局部规则命中 + 局部评分增强”

所以这份文件暴露出来的不是单个 bug，而是当前代码体系的几个短板：

1. 资格条件中的“行业错位资质”还不会抓
2. 评分项中的“内容错位”还不会抓
3. 标的域错贴识别还不够稳
4. 技术和商务边界判断在柴油发电机场景下触发不足
5. 区块切分还会带来后段噪声

## 八、下一步最值得补的点

结合这份文件的差异，最值得优先增强的是：

1. `资格错位证书规则`
   - 行业无关资质
   - 一般纳税额/经营年限/单项合同金额门槛

2. `评分内容错位识别`
   - 商务评分混入企业规模、资产、纳税额、成立年限
   - 方案评分混入案例和检测报告形式

3. `标的域与条款域不匹配识别`
   - 柴油发电机项目里出现 `有机产品认证`
   - 出现 `水运机电工程专项监理`
   - 出现与主标的明显不相称的资质或证书

4. `技术与商务场景化 LLM 任务`
   - 生产日期写死某一年
   - 制造商工程师限定
   - 尾款与运行期绑定
   - 采购人绝对免责

5. `误报抑制`
   - 履约反馈表中的 `优/良/中/差`
   - 评审程序章节中的泛化文本
