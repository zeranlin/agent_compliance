### 2. 评分表仍然是代码最强项，技术和商务虽然补起来了，但表达成熟度还不如人工

这次代码高风险主要集中在评分和模板错贴。  
人工则明显进一步看到了：

- 技术标准错位
- 履约担保和售后保证金
- 系统接口与设备维护外扩

这说明当前链路里，`scoring_structure_analyzer` 仍然最稳定；`technical_reference_consistency_engine` 和 `commercial_burden_analyzer` 已经起效，但在混合标的项目里的解释质量还不如人工自然。

### 3. 代码离人工主要差在“改稿组织能力”

- 资格门槛删什么

- 技术错贴怎么删
- 商务资金占用怎么调

代码现在虽然能输出主题，但更多还是“发现问题”。  
它还没有完全升级到“按改稿动作组织主问题”的程度。

### 4. 当前 `finding_arbiter` 已经更积极，但代表性证据和子主题拆分还可继续优化

这轮仲裁层已经把技术标准错位、保证金转换和资格整段问题上浮出来了。  
当前更需要继续增强的是：
- 主问题标题更像正式审查意见
- 同一主问题下证据摘录更代表性
- 混合标的场景下的子主题拆分更自然

## 六、代码审查下一步需要改进的设计和功能点

### 1. `qualification_bundle_analyzer` 需要继续增强“资格主问题里的改稿边界”

重点补：

- 成立年限
- 一般纳税门槛
- 人员规模门槛
- 资产规模门槛
- 异地经营场所门槛
- 错位行业资质

目标不是只把资格整段合起来，而是更稳定地区分：
- 法定可保留条件
- 应删除的一般门槛
- 明显错位的行业资质

### 2. `technical_reference_consistency_engine` 需要继续增强混合标的边界解释

重点补：

- 设备参数对应标准是否与标的真实相关
- 检测报告是否被限定为特定形式、特定机构、特定标准组合
- 药品采购与自动化设备参数之间的适配边界

目标是把技术章节稳定拆成：
- 标准引用错位
- 证明形式过严

### 3. `commercial_burden_analyzer` 需要继续增强“资金占用 + 义务外扩 + 响应责任”联动

重点补：

- 履约保证金比例
- 保证金形式限制
- 验收后转售后保证金
- 长质保期占压

目标是在现有资金占用主题基础上，进一步把：
- 售后响应
- 设备配套维护
- 额外服务义务

和资金占用分开表达。

### 4. `domain_match_engine` 需要支持混合采购场景的边界判断

这份文件不是单纯药品采购，而是：

- 药品
- 自动化发药设备
- 信息化接口
- 运维和清洁义务

所以下一步不是只抓“错位词”，而是要判断：
- 哪些是合理配套
- 哪些已经越过本次采购标的边界

### 5. `finding_arbiter` 需要继续往“正式审查意见化”推进

这次已经能出主题，但还需要继续做两件事：

- 正文与附件重复条款只保留一条主问题
- 同一主题下优先保留最适合改稿的代表性证据

这样结果会更像人工正式审查意见，而不是“主题已经对了，但表达还偏结构化”。

## 七、这次对比的结论性判断

一句话说，这份中药配方颗粒项目上，当前代码审查已经从“只会抓评分错位和模板残留”提升到“能稳定产出资格、评分、技术、商务主问题”，但仍弱于人工对：

- 资格条件整段异常
- 技术标准与证明形式错位
- 履约担保和售后保证金资金占用
- 药品采购与信息化/设备服务边界混合

这些章节级问题的整体把握。

所以这次差异的核心原因可以概括成三点：

1. 代码已经能做主题分析，但仍偏启发式归并，弱于人工的整段语义判断。  
2. 代码在评分表上的成熟度仍高于技术和商务混合场景。  
3. 代码结果已进入主题化阶段，但离“可直接用于改稿的人工式主问题”还差一层证据收束和改稿组织能力。
