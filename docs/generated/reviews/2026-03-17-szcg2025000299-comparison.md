## 五、为什么会出现这些差异

### 1. 当前代码更擅长抓“高频显眼异常”，人工更擅长读完整个资格段

这份文件最显眼的错位内容包括：

- `SPCA`
- `有害生物防制`
- `生活垃圾分类`
- `信息化端口无缝对接`

所以代码对这些高频错位词反应比较好。  
但资格条件里那些“年限、纳税、人员、资产、异地场所、棉花加工”是一个整组异常，需要 `qualification_bundle_analyzer` 做更强的整段分析，目前这层还不够。

### 2. 代码更容易在评分表起作用，技术和商务章节召回仍偏弱

这次代码高风险主要集中在评分和模板错贴。  
人工则明显进一步看到了：

- 技术标准错位
- 履约担保和售后保证金
- 系统接口与设备维护外扩

这说明当前链路里，`scoring_structure_analyzer` 和 `domain_match_engine` 已经比较有效，但 `technical_reference_consistency_engine` 和 `commercial_burden_analyzer` 还没有在这个样本上形成同等稳定召回。

### 3. 代码还不够像人工那样按“改稿动作”拆主题

人工会自然分成：

- 资格门槛删什么
- 评分结构怎么改
- 技术错贴怎么删
- 商务资金占用怎么调

代码现在虽然能输出主题，但更多还是“发现问题”。  
它还没有完全升级到“按改稿动作组织主问题”的程度。

### 4. 当前 `finding_arbiter` 仍偏保守

为避免误报，仲裁层这次更倾向保留已经确信的评分和模板错位问题，而没有把技术标准错位、保证金转换、系统接口外扩这些内容全部上浮成主问题。  
这让结果更稳，但也造成了对人工高价值问题的欠召回。

## 六、代码审查下一步需要改进的设计和功能点

### 1. `qualification_bundle_analyzer` 需要增强整段资格门槛归并

重点补：

- 成立年限
- 一般纳税门槛
- 人员规模门槛
- 资产规模门槛
- 异地经营场所门槛
- 错位行业资质

目标不是再多打一堆点，而是稳定长出一条：
`资格条件叠加设置一般门槛并混入错位资质`

### 2. `technical_reference_consistency_engine` 需要增强“标准错位 + 证明形式限制”双重识别

重点补：

- 设备参数对应标准是否与标的真实相关
- 检测报告是否被限定为特定形式、特定机构、特定标准组合
- 药品采购与自动化设备参数之间的适配边界

目标是把技术章节稳定拆成：
- 标准引用错位
- 证明形式过严

### 3. `commercial_burden_analyzer` 需要补“资金占用异常”主题

重点补：

- 履约保证金比例
- 保证金形式限制
- 验收后转售后保证金
- 长质保期占压

目标是稳定形成：
`履约担保和售后保证金安排导致异常资金占用`

### 4. `domain_match_engine` 需要支持混合采购场景的边界判断

这份文件不是单纯药品采购，而是：

- 药品
- 自动化发药设备
- 信息化接口
- 运维和清洁义务

所以下一步不是只抓“错位词”，而是要判断：
- 哪些是合理配套
- 哪些已经越过本次采购标的边界

### 5. `finding_arbiter` 需要继续压附件重复，并按改稿动作组织结果

这次已经能出主题，但还需要继续做两件事：

- 正文与附件重复条款只保留一条主问题
- 同一主题下优先保留最适合改稿的代表性证据

这样结果会更像人工正式审查意见，而不是结构化问题清单。

## 七、这次对比的结论性判断

一句话说，这份中药配方颗粒项目上，当前代码审查已经能稳定抓到评分错位、主观评分、模板残留和部分单方责任问题，但仍弱于人工对：

- 资格条件整段异常
- 技术标准与证明形式错位
- 履约担保和售后保证金资金占用
- 药品采购与信息化/设备服务边界混合

这些章节级问题的整体把握。

所以这次差异的核心原因可以概括成三点：

1. 代码更擅长高频错位词识别，弱于资格整段和技术整段综合判断。  
2. 代码在评分表上的召回好于技术和商务章节。  
3. 代码结果已开始主题化，但离“直接用于改稿的人工式主问题”还差一层章节级拆分和代表性证据收束。
