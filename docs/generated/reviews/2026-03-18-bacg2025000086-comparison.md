## 三、当前代码已经追上的部分

### 1. 评分结构主线已经基本追上人工

当前代码已经稳定上浮：

- `多个方案评分项大量使用主观分档且缺少量化锚点`
- `人员与团队评分混入错位证书并过度堆叠条件`

- `评分项名称、内容和评分证据之间不一致`
- `商务评分将企业背景和一般财务能力直接转化为高分优势`

这与人工审查对以下问题的判断方向是一致的：

- 方案评分主观性过强
- 团队评分里学历、证书、经验和荣誉堆叠
- 商务评分中体系认证、注册资本、营收和净利润直接折算高分
- 评分项名称与实际计分内容、证据不一致

### 2. 资格错位和模板残留已经被代码看见

当前代码已经稳定上浮：

并能把 `生活垃圾分类服务认证证书` 识别出来。说明当前 `domain_match_engine` 已能捕捉明显的跨领域内容错位。

### 3. 商务链条偏重已经进入章节级主问题

当前代码已经稳定上浮：

- `履约全链路中的付款、验收、责任和到场响应边界整体偏向供应商承担`
- `驻场、短时响应或服务场地要求形成事实上的属地倾斜`
- `验收送检、检测和专家评审费用整体转嫁给供应商`
- `验收程序、复检与最终确认边界不清`

这已经覆盖了人工审查里最核心的商务链条问题：

- 驻场和到场响应过强
- 付款与履约评价绑定
- 验收和专家评审边界不清
- 费用和风险偏向供应商承担

## 四、人工已发现但代码仍偏弱或仍有偏差的部分

### 1. 资格错位内容仍被定性得偏宽

人工会更明确地区分：

- 资格要求中错位的认证和治理类证书
- 评分项中的错位证书和认证

当前代码虽然抓到了 `生活垃圾分类服务认证证书`，但仍统一上浮成：

- `文件中存在与标的域不匹配的模板残留或义务外扩`

这比人工的表达更宽。人工会更倾向于写成：

- 资格条件中混入不匹配认证和治理证书

说明 `qualification_reasoning_engine` 和 `domain_match_engine` 之间，还需要把“资格错位”与“模板错贴”边界拉得更开。

### 2. 软件著作权高分问题已经被看见，但还没有完全写成最突出的评分主问题

人工认为本项目一个很核心的问题是：

- 自主知识产权评价对多类软件著作权每类给 20 分，最高 100 分

当前代码能从 `评分项名称、内容和评分证据之间不一致`、`单一评分因素权重设置过高` 中看见这组问题，但还没有把它稳定收束成一条更直接的主问题，例如：

- 软件著作权评分过高且与履约能力评价边界不清

说明 `scoring_semantic_consistency_engine` 对“评分主题漂移 + 高分值软件著作权”的识别还可以继续增强。

### 3. 经验评价和履约评价依赖既有合作证明的问题仍表达得不够聚焦

人工会明确指出：

- 经验评价要求国家机关或事业单位委托的同类项目
- 还要求甲方出具满意或优秀履约评价
- 且给到最高 100 分

当前代码已经通过多条 `单一评分因素权重设置过高` 和评分语义主题捕捉到了这组风险，但还没有把“经验 + 主观履约评价证明 + 高分值”稳定写成一条清晰主问题。

### 4. 商务链路主题仍有跨章节归并过宽的问题

当前代码的商务主问题：

- `履约全链路中的付款、验收、责任和到场响应边界整体偏向供应商承担`

方向是对的，但代表性证据覆盖了较长章节范围，仍然把项目管理要求、商务条款、合同违约责任等内容并得过大。人工通常会进一步拆成：

- 付款与履约评价绑定
- 驻场和响应要求偏重
- 违约责任和单方控制偏重
- 验收和专家评审边界不清

说明 `commercial_lifecycle_analyzer` 和 `finding_arbiter` 还需要继续收束主题边界。

## 五、为什么会出现这些差异

### 1. 当前代码更擅长抓“评分高频模式”，但对评分主题漂移的独立上浮还不够

这份文件的评分表问题很多，代码已经能看见：

- 主观评分
- 高权重认证
- 团队证书堆叠
- 注册资本和营收净利润打分
- 软件著作权高分

但还不够会把这些碎点重新组合成更像人工的少数主问题。尤其是“软件著作权高分”和“经验评价+满意评价证明”仍未形成最突出的独立主问题。

### 2. 资格错位、评分错位和模板错贴之间仍有边界重叠

这份文件同时出现了：

- 资格里的不匹配认证
- 评分里的错位证书
- 文件中的跨领域服务认证

人工会按章节边界分开写。当前代码虽然都能看见，但还更容易并到较宽的 `template_mismatch` 或 `scoring_content_mismatch` 主题里。

### 3. 商务链路分析已经成型，但仲裁层仍偏保守

代码已经具备了 `commercial_lifecycle_analyzer`，能把付款、驻场、责任、验收联起来看；但 `finding_arbiter` 目前还偏保守，更愿意保留大主题，尚未完全收敛到人工改稿时常用的几条商务主问题。

## 六、代码审查下一步需要改进的设计和功能点

### 1. 继续增强 `qualification_reasoning_engine`

重点补：

- 把资格错位认证、治理评级、管理体系与模板残留区分开
- 让“资格错位”稳定上浮为资格章节主问题，而不是统一并入模板错贴

### 2. 继续增强 `scoring_semantic_consistency_engine`

重点补：

- 把“软件著作权高分 + 评分目的漂移 + 高权重结构”稳定合成为独立主问题
- 把“经验评价 + 履约评价证明 + 高分值”合成为独立主问题

### 3. 继续增强 `commercial_lifecycle_analyzer`

重点补：

- 把付款、履约评价、驻场响应、违约后果和验收程序拆成更利于改稿的商务子主题
- 避免一个商务大主题覆盖过多章节和证据

### 4. 继续增强 `finding_arbiter`

重点补：

- 在保持章节级主问题的前提下压缩过宽的跨章节归并
- 为每条主问题选择更短、更准的代表性证据
- 让输出更像人工正式审查意见，而不是“主题正确但表达仍偏粗”
