## 三、当前代码已经追上的部分

### 1. 评分结构与评分内容错位主线已经被看见

当前代码已经稳定上浮：

- `评分项名称、内容和评分证据之间不一致`
- `商务评分将企业背景和一般财务能力直接转化为高分优势`
- `认证评分混入错位证书且高分值结构失衡`
- `评分分档缺少明确量化锚点`
- `单一评分因素权重设置过高`

这和人工审查的主线是吻合的：

- 绿色采购承诺和物业类荣誉被混进评分；
- 同类业绩、注册资本、营业收入、净利润被叠加成高分优势；
- 体系认证权重过高；
- 主观评分分档缺少量化锚点。

### 2. 商务链条偏重已经形成章节级主问题

当前代码已经稳定上浮：

- `驻场、短时响应或服务场地要求形成事实上的属地倾斜`
- `履约全链路中的付款、验收、责任和到场响应边界整体偏向供应商承担`
- `验收程序、复检与最终确认边界不清`

这已经覆盖了人工审查里最核心的商务问题：

- `2小时响应 + 12小时到场 + 48小时修复 + 免费备用机` 的组合要求偏重；
- 财政支付延期免责、争议款项暂停支付、第三方检测不合格即单方解除等条款整体压重供应商责任；
- 验收和送检边界不清。

### 3. 技术章节中的“标准错位 + 证明形式过严”已经形成强主问题

当前代码已经能稳定上浮：

- `技术证明材料形式要求过严且带有地方化限制`

- `ISO 10993`
- `经广告审查机关备案的产品彩页`
- `CMA 报告 + 全国认证认可信息公共服务平台截图`

这类问题收束成更明确的技术主问题。

### 4. 混合采购边界主问题已经被稳定上浮

当前代码已经能稳定上浮：

- `设备采购场景叠加信息化接口和碳足迹义务，边界不清`

这条主问题已经覆盖了人工审查里高度关注的这些内容：

## 四、人工已发现但代码仍偏弱或缺失的部分

### 1. 资格主问题仍偏“总括”，拆分粒度还不如人工自然

人工式审查会更自然地拆成：

- `农民专业合作社不具备投标资格`
- `高空清洗悬吊作业企业安全生产证书`
- `高新技术企业证书`
- `企业诚信管理体系认证证书`

当前代码虽然已经补出了：

- `资格条件整体超出法定准入和履约必需范围`
- `资格条件中存在与标的域不匹配的行业资质或专门许可`

但这两条之间还存在一定重叠，距离人工那种“整体结论 + 明确子项”的改稿感还差一步。

### 2. 商务链路主题仍偏宽

人工更容易继续拆成：

- 到场响应和备用机要求偏重
- 第三方检测、送检和复验费用转嫁
- 单方解除和责任后果偏重
- 付款与验收/整改边界不够稳

当前代码现在主要还是：

- `履约全链路中的付款、验收、责任和到场响应边界整体偏向供应商承担`
- `验收程序、复检与最终确认边界不清`

方向是对的，但仍不如人工那样拆得利于直接改稿。

## 五、为什么会出现这些差异

### 1. 资格分析器已经补上召回，但主题拆分仍偏保守

这轮 `qualification_reasoning_engine` 和资格错位分析器已经把漏判补上了，但为了避免误拆，目前仍偏向先给“整体超出法定准入范围”的总括主题，再补“行业资质错位”子主题。

### 2. 技术分析器已经成形，但“需论证”层仍会保留一些中风险补充点

当前 `technical_reference_consistency_engine` 已经能稳定拆出：

- 标准错位
- 证明形式过严

但 `technical_necessity_explainer` 仍会保留若干中风险“需论证”点，所以技术章节结果还没有人工那样克制。

### 3. 混合采购边界理解已经明显增强，但还需继续压缩补充噪声

当前 `mixed_scope_boundary_engine` 已经能把设备接口和碳足迹义务上浮成主问题，但对同一章内其他扩展义务的补充点仍然偏多。

### 4. 商务生命周期分析器仍以“大主题 + 补充子问题”为主

当前 `commercial_lifecycle_analyzer` 已经把方向拉准，但 `finding_arbiter` 还没有完全把商务大主题拆成更适合改稿的几个细分主题。

这和人工输出风格还有差距。

## 六、代码审查下一步需要改进的设计和功能点

### 1. 继续增强 `qualification_reasoning_engine`

重点补：

- 在保持整体资格主题的同时，把“组织类型限制”“行业资质错位”“一般规模门槛”拆得更利于改稿；
- 避免总括主题和子主题之间的证据重叠过多。

### 2. 继续增强 `technical_reference_consistency_engine`

重点补：

- 继续压缩“需论证”补充问题数量；
- 让“标准错位”“证明形式过严”两条主问题的代表性摘录更短、更准；
- 对 `CMA报告 + 平台截图 + 专项检测` 组合要求继续增强场景判断。

### 3. 继续增强 `mixed_scope_boundary_engine`

重点补：

- 继续识别“设备采购 + 软件接口 + 医院信息化对接”的合理边界；
- 进一步吸收同章的接口、数据交换和 ESG 扩展义务碎点；
- 保持混合场景主问题，不再回退成泛化模板主题。

### 4. 继续增强 `commercial_lifecycle_analyzer`

重点补：

- 把售后响应、备用机、第三方检测、费用承担、单方解除和责任后果继续分层；
- 让商务主问题更接近采购人可直接改稿的结构。

### 5. 继续增强 `finding_arbiter`

重点补：

- 继续压掉中风险补充点里的重复技术论证主题；
- 为每条主问题选择更短、更准的代表性证据；
- 减少“方向对，但主题过大”的情况。
