## 三、当前代码已经追上的部分

- `商务评分将企业背景和一般财务能力直接转化为高分优势`

这和人工审查的主线是吻合的：

- 同类业绩、注册资本、营业收入、净利润被叠加成高分优势；
- 体系认证权重过高；
- 主观评分分档缺少量化锚点。

### 2. 商务链条偏重已经形成章节级主问题

当前代码已经稳定上浮：

- `驻场、短时响应或服务场地要求形成事实上的属地倾斜`
- `履约全链路中的付款、验收、责任和到场响应边界整体偏向供应商承担`
- `验收程序、复检与最终确认边界不清`

这已经覆盖了人工审查里最核心的商务问题：

- `2小时响应 + 12小时到场 + 48小时修复 + 免费备用机` 的组合要求偏重；
- 财政支付延期免责、争议款项暂停支付、第三方检测不合格即单方解除等条款整体压重供应商责任；
- 验收和送检边界不清。

### 3. 技术条款中的证明形式过严已经有一定感知

当前代码已经能上浮：

- `检测证明形式要求可能合理但需补充必要性论证`
- `技术要求可能合理但需补充必要性论证`

说明当前 `technical_reference_consistency_engine` 和 `technical_necessity_explainer` 已经能对 `CMA 报告 + 平台截图 + 第三方专项检测` 这类要求发出预警。

## 四、人工已发现但代码仍偏弱或缺失的部分

### 1. 资格错位条款基本漏掉了

人工式审查明确指出：

- `农民专业合作社不具备投标资格`
- `高空清洗悬吊作业企业安全生产证书`
- `高新技术企业证书`
- `企业诚信管理体系认证证书`

这些都与血透类设备供货履约没有直接关系，属于明显的资格错位或不当扩大准入门槛。

当前代码没有把这一组问题稳定上浮成资格章节主问题。  
这说明 `qualification_reasoning_engine` 在“医疗设备项目里的错位资格识别”上仍明显偏弱。

### 2. 技术章节里的“标准错位”和“证明形式过严”还没有拆成强主问题

人工会明确指出：

- `EN14175-3:2019` 与血透设备场景适配性存疑；
- `ISO 20743抗菌 + ISO 10993` 与多个参数一起被写进检测要求；
- 大量参数都要求 `CMA报告 + 全国认证认可信息公共服务平台截图 + 不在许可范围即不符合`；
- 机身尺寸等还强制要求专项检测报告。

当前代码虽然给出了“需论证”和“证明形式要求可能合理但需补充必要性论证”，但力度仍偏软，没像人工那样把：

- `标准错位`
- `证明形式过严`

拆成两个足够突出的技术主问题。

### 3. 混合采购边界问题还没有被稳定上浮

人工认为这份文件里有明显的附加义务外扩：

- 免费开放软件端口
- 无偿配合与医院 HIS/PACS/LIS 对接
- 数据交换和对接费用全部由供应商承担
- 30天内提供并持续优化碳足迹盘查报告

当前代码只上浮了一条：

- `条款内容可能存在模板错贴或义务外扩`

但没有更明确地区分：

- 设备采购中的信息化接口边界
- 与主标的无直接关系的碳足迹义务

说明 `mixed_scope_boundary_engine` 和 `domain_match_engine` 在“设备采购 + 医院信息化接口 + ESG类附加义务”这种混合场景上还不够强。

### 4. 商务链条主题仍然偏宽

当前代码的商务主问题方向是对的，但仍然把：

- 付款规则
- 售后响应
- 第三方检测
- 责任承担
- 合同违约后果

收得过大。人工更容易继续拆成：

- 到场和备用机要求偏重
- 付款免责和暂停支付偏重
- 检测、送检、复验费用转嫁
- 单方解除和违约责任偏重

说明 `commercial_lifecycle_analyzer` 和 `finding_arbiter` 还需要继续分层。

## 五、为什么会出现这些差异

### 1. 当前代码对“医疗设备项目中的资格错位”规则覆盖还不够

这份文件的资格错位不是普通的注册资本、营收门槛，而是：

- 高空清洗
- 高新技术企业
- 企业诚信管理体系
- 农民专业合作社排除

这类条款更像“行业错位资质 + 组织类型不当限制”的组合。当前代码对这种组合的识别还不够系统。

### 2. 技术条款分析更偏“需论证”而不是“直接指出问题结构”

代码已经能感知到第三方检测、证明形式和部分技术条款有问题，但更倾向于保守标成：

- `需论证`
- `证明形式可能过严`

而人工会更直接地把这类问题拆成：

- 标准引用与标的不匹配
- 证明形式和取证路径过严

### 3. 混合采购场景边界理解还弱于人工

人工能更直接地区分：

- 真正属于设备联机和数据接口的必要配套要求
- 超出本项目范围的信息化或碳足迹义务

当前代码还更容易把它们统一压成 `template_mismatch` 或一般性的义务外扩。

### 4. 仲裁层仍偏保守

当前代码已经能形成章节主问题，但在商务和技术部分仍偏保守：

- 宁可保留偏宽的主主题
- 不够积极地往“少数可改稿子问题”继续拆

这和人工输出风格还有差距。

## 六、代码审查下一步需要改进的设计和功能点

### 1. 继续增强 `qualification_reasoning_engine`

重点补：

- 医疗设备采购场景下的资格错位资质识别；
- 对 `高空清洗`、`高新技术企业`、`诚信管理体系`、`农民专业合作社排除` 这类条款形成稳定的资格主问题；
- 区分“错位资质”“组织类型限制”“一般财务/规模门槛”。

### 2. 继续增强 `technical_reference_consistency_engine`

重点补：

- 把 `标准错位` 和 `证明形式过严` 稳定拆成两条技术主问题；
- 对 `EN14175-3`、`ISO 20743`、`ISO 10993` 这类引用做更强的场景适配判断；
- 对 `CMA报告 + 平台截图 + 专项检测` 组合要求形成更强结论。

### 3. 继续增强 `mixed_scope_boundary_engine`

重点补：

- 识别“设备采购 + 软件接口 + 医院信息化对接”的合理边界；
- 识别“碳足迹盘查、持续优化”这类超出主标的范围的附加义务；
- 形成比 `template_mismatch` 更具体的混合场景主问题。

### 4. 继续增强 `commercial_lifecycle_analyzer`

重点补：

- 把售后响应、备用机、付款免责、暂停支付、第三方检测、单方解除继续分层；
- 让商务主问题更利于采购人直接改稿。

### 5. 继续增强 `finding_arbiter`

重点补：

- 把技术和商务主题继续收束成更少、更利于改稿的主问题；
- 为每条主问题选择更短、更准的代表性证据；
- 减少“方向对，但主题过大”的情况。
