# [BACG2025000091-A]深圳市宝安区中心医院三维电生理导航系统采购项目

## 人工审查 vs 代码审查对比

### 一、总体结论

- 人工式审查结论：`高风险，不建议按现文本直接发标`
- 当前代码审查结果：`9` 条 findings，其中高风险 `7` 条、中风险 `2` 条

本轮增强后，代码结果已经明显更接近人工主线，尤其在评分结构、技术证明形式、商务链路和混合采购边界这四类问题上已经能够稳定长出章节级主问题。

### 二、当前已经基本对齐的主问题

1. 评分主题漂移
- 人工：指出售后服务评分混入“先进单位”荣誉、注册安全工程师证书，并叠加高分值主观分档。
- 代码：已上浮 `评分项名称、内容和评分证据之间不一致`，能覆盖荣誉、证书和服务方案错位问题。

2. 主观评分分档过重
- 人工：指出多个方案评分项使用高分值主观分档。
- 代码：已上浮 `多个方案评分项大量使用主观分档且缺少量化锚点`。

3. 免费质保期高权重评分
- 人工：指出每延长一年免费质保期即给100分，明显失衡。
- 代码：已上浮 `免费质保期延长按年度直接高分赋值`。

4. 技术证明形式过严
- 人工：指出广告审查备案彩页、CMA/CNAS、认e云截图、专项检测报告等路径过严，并有“投标资格必备文件”倾向。
- 代码：已上浮 `技术证明材料形式要求过严且带有地方化限制`。

5. 商务责任和支付链条偏重
- 人工：指出响应、备用机、开机率、财政审批免责、暂停支付和第三方检测后果整体偏向采购人。
- 代码：已上浮 `履约全链路中的付款、验收、责任和到场响应边界整体偏向供应商承担`。

6. 混合采购场景边界外扩
- 人工：指出设备采购叠加 HIS/PACS/LIS 对接和碳足迹义务。
- 代码：已上浮 `设备采购场景叠加信息化接口和碳足迹义务，边界不清`。

### 三、当前仍弱于人工的地方

1. 评分问题的拆分还不够“改稿化”
- 代码已经能看见售后服务评分错位，但当前还是用较泛的 `评分项名称、内容和评分证据之间不一致` 来承载。
- 人工会更自然地拆成：
  - 售后服务方案评分混入荣誉和证书
  - 质保期延长高权重评分
  - 方案项高分值主观分档

2. 技术证明条款中的“准入化倾向”还没单独上浮
- 第2.5项将专项检测报告表述为“投标资格必备文件”，人工会把它看成技术证明路径被实质前置为门槛。
- 当前代码仍将其并在“技术证明材料形式要求过严”主题里，没有再拆成更强的准入风险。

3. 商务链路主题仍偏大
- 当前代码已经收到了履约全链路主问题，但人工仍会继续细分：
  - 响应/备用机/开机率责任过重
  - 付款免责和暂停支付过宽
  - 第三方检测和解除合同边界不清

### 四、本轮暴露并已开发的能力增强点

本轮不是只做对比，还已经把当前文件暴露的能力缺口先补进了代码：

1. 强化 `scoring_semantic_consistency_engine`
- 新增对 `先进单位`、`注册安全工程师` 等服务主题错位评分内容的识别。

2. 新增高频显性评分规则
- 新增 `免费质保期延长按年度直接高分赋值` 的显性规则和主题上浮能力。

3. 强化 `commercial_lifecycle_analyzer`
- 将 `2小时响应`、`12小时到场`、`48小时修复`、`备用设备`、`年开机率`、`财政审批免责`、`暂停支付` 纳入全链路主题识别。

4. 强化 `finding_arbiter` 和噪声过滤
- 让两条高分值主观方案项也能上浮成章节主问题。
- 压掉“绿色采购包装承诺”“符合国家有关安全环保卫生规定”这类低价值技术误报。
- 让验收边界主题不再串入评分条款。

### 五、下一轮最值得继续补的功能点

1. `service_scheme_scoring_analyzer`
- 将售后服务方案中的荣誉、证书和主观分档稳定拆成独立主问题，而不是继续挂在通用评分语义主题下。

2. `technical_proof_as_admission_barrier_engine`
- 专门识别“专项检测报告”“原件扫描件”“某平台截图”等技术证明材料被实质前置为投标资格或投标必备文件的情形。

3. `commercial_lifecycle_analyzer` 二阶段
- 将当前过大的商务全链路主题继续拆成：
  - 到场响应/备用机/开机率责任
  - 付款免责和暂停支付
  - 第三方检测与解除合同边界

4. `evidence_selector` 二阶段
- 进一步压缩代表性证据，让每条主问题都更像人工会直接引用给业务方改稿的那一段。

### 六、当前判断

这份文件上，代码审查已经从“只能抓到少量碎点”进步到“能够抓住评分、技术、商务和边界外扩四条主线”。当前剩余差距已经不主要是召回，而是：

- 评分问题拆分是否足够利于改稿
- 技术证明是否能进一步识别成“实质准入门槛”
- 商务链条能否从一个大主题继续拆成更清楚的三个子主题

这说明本轮闭环有效：当前文件暴露出的能力缺口，已经先转化成代码增强，再进入下一份文件。
