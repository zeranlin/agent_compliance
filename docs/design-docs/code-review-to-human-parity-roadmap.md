# 代码审查逼近人工审查的增强路线

## 目标

把当前本地代码审查能力，从“离线初筛器”逐步增强为“规则、结构分析、知识检索、局部模型推理、评测闭环”协同工作的审查系统，使其持续逼近人工审查效果，而不是只在高频显性问题上起作用。

本路线的核心目标不是替代人工法务或采购复核，而是让代码审查在以下维度持续接近人工：

- 问题识别覆盖率
- 复合问题拆分能力
- 边界问题判断能力
- 定位与证据链完整度
- 修改建议可用性
- 复审一致性

## 一、当前差距的本质

基于窗帘采购项目对比样例可见，当前代码与人工审查的差距主要体现在：

1. 问题拆分过粗  
   一段资格条件中存在多个独立高风险点时，代码容易只产出一个泛化 finding。

2. 评分结构理解不足  
   代码能识别部分荣誉、证照、资格重复计分问题，但对“认证高权重”“业绩高权重”“样品主观评分过重”等结构性失衡识别不足。

3. 技术参数边界判断不足  
   对医疗、医院场景中可能有正当性的高参数要求，当前代码偏保守，缺少“有合理性但仍需论证”的中间判断层。

4. 商务责任和付款条款识别不足  
   代码已经能抓模板残留和验收边界，但还不擅长识别绝对免责、无限扩大责任、抽检和付款联动这类复合风险。

5. 上下文联动不足  
   人工审查会把“资格-评分-样品-商务-付款”几个部分联动起来判断，而当前代码仍偏向单条命中。

## 二、总体架构升级方向

建议把现有离线执行引擎继续升级为 6 层协同结构：

1. 文档标准化层  
2. 规则与模式识别层  
3. 结构分析层  
4. 本地知识检索层  
5. 局部模型推理层  
6. 评测与迭代层

一句话概括：

`规则找候选问题 -> 结构分析判断上下文关系 -> 检索给依据 -> 局部模型做边界判断 -> 固化为结构化 findings -> benchmark 反向迭代`

## 三、必须补的功能模块

### 1. 资格条件细拆引擎

目标：
- 把一段资格要求中的多个异常门槛拆成多个独立 finding，而不是只输出一条泛化结论。

重点识别：
- 上级主管单位同意函
- 所有制/股权结构/国资比例
- 注册资本、营收、利润、经营年限
- 连续多年审计报告
- 国家级特色企业、示范企业、专精特新等企业属性标签

建议实现：
- 基于规则词典与模式切分先定位疑似门槛
- 对同一段资格条款按“语义断点 + 分号/句号 + 门槛类型”做细拆
- 对拆分结果做轻量合并，避免碎片化过头

### 2. 评分结构分析器

目标：
- 不是只看评分文字，而是理解评分项在整套评分结构里的角色。

要输出的结构化信息：
- 评分项类别：方案、样品、认证、业绩、价格、商务、信用、售后等
- 权重和分值
- 是否主观评分
- 是否混入资格材料
- 是否混入与履约弱相关因素
- 是否存在中标后补证、承诺替代现时证明

重点能力：
- 识别单项分值是否过高
- 识别同类因素总分是否过高
- 识别样品评分是否主观锚点不足
- 识别认证证书评分是否结构性偏高

### 3. 商务责任与付款联动分析器

目标：
- 补足当前对商务条款的理解不足。

重点识别：
- 采购人绝对免责
- 供应商承担“一切事故”“全部责任”
- 损失及责任均与采购人无关
- 外部检测/终验/财政资金与付款条件深度绑定
- 验收、抽检、复检、返工、付款之间的联动关系

建议实现：
- 先通过规则标出责任和付款触发语句
- 再做局部上下文拼接，形成“责任条款组”“付款条款组”“验收条款组”
- 最后由代码或模型判断是否构成失衡

### 4. 技术参数边界判断器

目标：
- 解决“技术要求合理性”和“参数组合过窄”之间的边界问题。

特别适用：
- 医疗设备
- 医院布草/窗帘/隔帘
- 检测报告、性能参数、功能组合要求

建议输出 3 档结果：
- 明显合理
- 明显过窄/疑似定向
- 有场景正当性，但需要市场调研或必要性论证

建议实现：
- 规则负责识别高参数叠加、报告时段限制、检测报告资质要求
- 本地模型负责判断“是否属于场景必要但仍需论证”

### 5. finding 拆分与合并决策器

目标：
- 让代码输出的粒度更接近人工式审查。

需要避免两种错误：
- 合并过头：多个独立问题被压成一条
- 拆分过度：同一问题被切成很多碎片

建议策略：
- 资格条件：优先细拆
- 评分项：按评分因素归组
- 技术参数：同类参数跨章节可归并
- 商务条款：责任、验收、付款分别建组

### 6. 例外与正当性机制

目标：
- 避免代码把所有高要求都判成违规。

需要支持的输出：
- 风险存在
- 但可能具有业务或场景正当性
- 需采购人补充市场调研、必要性说明、临床/场景论证

这是逼近人工判断的关键，因为人工强项不只是“会指出问题”，还会知道“哪些问题不能说死”。

## 四、大模型应如何参与

目标不是“把整份文件交给大模型”，而是让大模型在最值钱的环节参与。

### 不建议交给大模型的任务

- 文档抽取
- 章节定位
- 高频硬规则扫描
- 基础知识索引
- 全文自由审查

### 建议交给大模型的任务

1. 边界判断  
   例如技术参数是否可能具有场景正当性，商务条款是否属于合理内控边界。

2. 复合问题拆分  
   例如一段资格要求应拆成几条 finding。

3. 上下文联动判断  
   例如一条评分项是否同时包含方案评价、本地业绩、荣誉、营业执照等多个不当因素。

4. 建议改写增强  
   在不改变结论的前提下，把修改建议和替代表述写得更接近人工审查意见。

### 推荐的参与强度

- 低强度：润色标题、风险说明、修改建议
- 中强度：边界判断、拆分判断
- 高强度：同一评分项/商务条款组的局部多条款联合推理

推荐路线：
- 70% 靠规则、结构分析、检索
- 30% 靠模型做局部语义判断

## 五、如何实现自动持续迭代

要让代码审查持续逼近人工审查，必须建立“差异 -> 规则/模型增强 -> 回归评测”的闭环。

### 1. 差异样本沉淀

每次人工审查与代码审查出现差异时，都应沉淀为结构化样本：

- 文件类型
- 人工多识别出的风险
- 代码误报点
- 代码漏报点
- 原因标签
- 建议补强方向

### 2. 差异标签体系

建议新增以下标签：
- `missed_qualification_split`
- `missed_scoring_weight_issue`
- `missed_post_award_proof_issue`
- `missed_one_sided_liability`
- `missed_payment_acceptance_link`
- `missed_technical_justification_boundary`
- `over_merged_findings`
- `over_fragmented_findings`

### 3. benchmark 回归

每次新增规则或模型策略后，都跑固定样例回归，验证：
- 是否补上漏报
- 是否引入新的误报
- 是否让 findings 粒度更接近人工

### 4. 版本化管理

建议对以下对象做版本化：
- `rule_set_version`
- `structure_analyzer_version`
- `reference_snapshot_id`
- `llm_policy_version`
- `finding_merge_policy_version`

这样才能解释“为什么这次结果和上次不同”。

## 六、建议的分阶段实施顺序

### 阶段 A：规则与结构补强

目标：
- 先补人工稳定能识别的硬风险，不依赖模型。

优先做：
- 资格条件细拆规则
- 评分高权重规则
- 中标后补证规则
- 商务责任失衡规则
- 抽检与付款联动规则

完成标志：
- 在典型货物采购文件中，人工显性高风险问题的命中率明显提升

### 阶段 B：局部结构分析

目标：
- 让系统从“单条命中”升级为“理解评分项、资格段、商务组”。

优先做：
- 评分项分类
- 分值占比分析
- 同段资格要求细拆
- 商务条款组装

完成标志：
- 同一评分项中的多个问题能更稳定拆分或归并

### 阶段 C：局部模型推理

目标：
- 用本地模型补足规则难以稳定覆盖的边界判断。

优先做：
- 技术参数边界判断
- 商务责任边界判断
- 评分结构性失衡的局部推理
- finding 拆分/合并建议

完成标志：
- 模型调用数量受控，但对复杂文件的质量有明显提升

### 阶段 D：持续评测与自动迭代

目标：
- 建立持续逼近人工审查的长期机制。

优先做：
- 差异样本入库
- benchmark 管理
- 误报漏报报告
- 版本对比报告

完成标志：
- 每轮增强都有明确收益，而不是凭感觉“好像更智能”

## 七、当前最值得马上做的 10 个开发任务

1. 新增资格条件细拆规则包
2. 新增评分高权重识别规则包
3. 新增“中标后补证/承诺替代现时证明”规则
4. 新增“绝对免责/一切责任/损失与采购人无关”规则
5. 新增“终验款与抽检绑定”规则
6. 为评分项增加结构化分类字段
7. 为商务条款增加责任组、验收组、付款组聚合
8. 把技术参数问题接入局部模型边界判断
9. 增加“人工 vs 代码差异样本”目录和格式
10. 把差异样本纳入 benchmark 回归

## 八、一句话结论

要让代码审查持续逼近人工审查，关键不是“把模型接得更深”，而是：

`把规则细拆、结构分析、知识检索、局部模型判断和 benchmark 迭代整合成一套可持续演进的审查系统。`
