# 采购需求风险识别优先级清单

## 目标

将当前代码审查路线进一步压缩为一条更聚焦的主线：

- 只优先增强“发现采购需求风险”的能力
- 不再把页面样式、导出花样、发布后裁判能力混入当前主线
- 所有后续增强，都优先回答一个问题：
  - **这项改动是否能更早、更准、更稳地发现采购需求中的风险**

## 当前第一原则

当前系统定位为：

- **采购需求合规性审查智能体**

当前最优先服务的使用场景是：

- 采购需求形成与发布前审查
- 帮助采购人在正式发布前识别风险、改稿和复核

因此，当前“能力优先级”应服从以下顺序：

1. 查得准
2. 查得全
3. 查得稳
4. 结论易改稿
5. 结论易复核

## 不再作为当前主线的事项

以下事项不是完全不做，而是**不再占用当前主线优先级**：

- 页面风格和展示层继续扩张
- 更复杂的导出模板
- 发布后的争议裁判能力
- 履约归责与责任认定
- 非采购需求阶段的流程能力扩张

## 当前最值得聚焦的 4 组 engine

### 1. 品目识别与场景理解 engine

目标：

- 更准确判断“采购对象到底是什么”
- 更稳定识别主品目、次品目和混合采购
- 更准确区分：
  - 合理配套
  - 轻度边界外扩
  - 明显错位要求

当前应继续增强的点：

- `procurement_catalog_classifier`
- `catalog_knowledge_profile`
- `mixed_scope_boundary_engine`
- `domain_match_engine`

高优先级误判点：

- 服务类文件外壳把货物项目带偏
- 智能化/信息化扩展把货物项目误识别成数字化服务项目
- 模板错贴与轻量配套义务区分不稳

### 2. 评分语义与结构失衡识别 engine

目标：

- 更稳定发现评分结构失衡
- 更稳定发现评分项名称、内容、证据、目的不一致
- 更稳定识别高权重评分项对中标结果的过度影响

当前应继续增强的点：

- `scoring_semantic_consistency_engine`
- `brand_and_certification_scoring_analyzer`
- `personnel_certificate_mismatch_engine`
- `demo_mechanism_engine`

高优先级误判点：

- 技术评分权重过高但未上浮
- 团队/证书/经验堆叠没有稳定收成独立主问题
- 评分错位被错误并到模板错贴或泛化问题

### 3. 混合边界与错位要求识别 engine

目标：

- 更稳定抓到跨领域模板残留
- 更稳定识别与当前采购标的不匹配的资质、认证、标准、接口和附加义务
- 更稳定区分“需论证”与“明显错位”

当前应继续增强的点：

- `mixed_scope_boundary_engine`
- `qualification_reasoning_engine`
- `technical_reference_consistency_engine`
- `catalog_knowledge_profile`

高优先级误判点：

- 药品 + 自动化设备 + 信息化接口
- 家具 + 资产定位/智能芯片
- 体育器材 + 轻量智能化功能
- 标识标牌/印刷 + IT/保安/软件著作权

### 4. 商务/验收链条风险识别 engine

目标：

- 更稳定识别付款、考核、扣罚、整改、验收、复检、解除合同之间的风险链条
- 更稳定识别资金占用、责任失衡、验收费转嫁和验收边界不清
- 更稳定把宽泛商务主题拆成采购人可直接改稿的子问题

当前应继续增强的点：

- `commercial_lifecycle_analyzer`
- `finding_arbiter`
- `confidence_calibrator`
- `rewrite_generator`

高优先级误判点：

- 付款与考核绑定没有稳定上浮
- 商务主题过宽，跨章节误并
- 费用转嫁和责任失衡被宽泛总主题吞掉

## 当前优先级排序

### P0

- 品目识别与场景理解 engine
- 评分语义与结构失衡识别 engine

原因：

- 这两层最直接决定“有没有看准项目”和“有没有看准评分风险”
- 当前多数误判和漏判，最早都是从这两层开始放大

### P1

- 混合边界与错位要求识别 engine
- 商务/验收链条风险识别 engine

原因：

- 这两层更依赖前面品目和评分理解的稳定性
- 但一旦前两层站稳，它们会显著提升整体接近人工审查的程度

## 后续每轮增强的判断标准

任何后续改动，优先用下面 5 个问题判断值不值得做：

1. 是否提升了品目识别准确率
2. 是否提升了评分主问题识别准确率
3. 是否降低了模板错贴/边界问题误判
4. 是否让商务和验收链条更容易直接改稿
5. 是否减少了主问题漂移、过碎或过宽

如果答案大多是否定的，这项改动就不应占用当前主线优先级。

## 当前推荐推进顺序

1. 继续增强 `catalog_knowledge_profile`
2. 继续增强 `scoring_semantic_consistency_engine`
3. 继续增强 `mixed_scope_boundary_engine`
4. 继续增强 `commercial_lifecycle_analyzer`
5. 继续增强 `finding_arbiter`
6. 再让 `confidence_calibrator` 和 `rewrite_generator` 配合收束输出

## 一句话结论

当前主线不再是“继续加更多功能”，而是：

- **围绕采购需求风险识别，持续把品目、评分、边界、商务四组 engine 调准。**

只有能直接提升这四组 engine 风险识别准确性的改动，才应占用当前最高优先级。
