# 人工式审查查点矩阵

## 目标

把“代码审查逼近人工审查”拆成稳定的查点矩阵，先回答两个问题：

1. 人工审查通常会看哪些查点
2. 当前代码在哪些查点上已经覆盖、哪些还明显缺口

本矩阵强调“先把问题找全”，而不是先追求精确算分或表达润色。

## 一、一级查点

当前建议把政府采购需求审查拆成 7 个一级查点：

1. 资格条件
2. 评分标准
3. 技术要求
4. 商务条款
5. 验收与检测
6. 付款与违约
7. 模板残留与义务外扩

## 二、二级查点与当前覆盖情况

### 1. 资格条件

人工重点会看：
- 是否出现所有制、股权结构、国资比例、企业性质限制
- 是否出现主管单位同意函、备案证明、属地要求
- 是否出现营收、利润、注册资本、经营年限等一般性财务门槛
- 是否要求连续多年审计报告
- 是否要求国家级特色企业、专精特新、高新技术企业等企业属性
- 是否把类似业绩、奖项荣誉、认证资质错误前置为准入门槛

当前代码状态：
- 已覆盖：
  - 所有制/股权结构
  - 主管单位同意函
  - 一般性财务门槛
  - 连续多年审计
  - 国家级特色企业等属性门槛
- 仍待补：
  - 属地备案/本地机构/本地服务团队前置门槛
  - 人员画像、年龄、性别、户籍等不当资格条件
  - 类似业绩被直接写成资格前置
  - 资格条件与法定资质、履约能力之间的“正当性例外”判断

### 2. 评分标准

人工重点会看：
- 评分项是否把营业执照、资格材料、荣誉证书重复转评分
- 评分项是否混入与履约弱相关的奖项、荣誉、信用等级、科技型中小企业等
- 样品评分、方案评分是否主观性过强
- 认证评分、业绩评分、样品评分是否占分过高
- 同一评分项中是否混入本地业绩、营业执照、守重证书等不同性质因素
- 是否存在中标后补证、承诺替代现时证明
- 是否存在评分结构整体失衡

当前代码状态：
- 已覆盖：
  - 资格重复计分
  - 荣誉资质加分
  - 主观分档
  - 样品高分
  - 认证高分
  - 业绩高分
  - 中标后补证
  - 评分结构整体失衡的综合 finding
- 仍待补：
  - 评分表总分/权重解析
  - 不同评分项之间的累计占分分析
  - 方案评分、售后评分、团队评分等更多类别识别
  - “同一评分项混入多个不同性质因素”的结构化拆分
  - 价格分、商务分、技术分之间的整体平衡性判断

### 3. 技术要求

人工重点会看：
- 是否存在品牌、型号、专利、专有平台、专有接口定向
- 是否存在参数区间过窄或多参数叠加锁定少数产品
- 是否存在报告时段、检测机构、认证资质要求过严
- 是否存在兼容性要求但未说明必要性
- 是否存在场景上可能合理但仍需市场调研或必要性论证的边界条款

当前代码状态：
- 已覆盖：
  - 部分定向/过窄参数识别
  - 同类技术参数跨章节归并
- 仍待补：
  - 品牌/型号/平台类更强规则
  - 参数组合叠加判断
  - 技术参数正当性与必要性边界判断
  - 检测报告时段、机构资质、证据形式限制
  - 需本地模型参与的“合理但需论证”中间层判断

### 4. 商务条款

人工重点会看：
- 是否存在采购人绝对免责
- 是否存在“一切事故全部由供应商承担”“负全责”之类责任失衡
- 是否存在财政审批、资金到位、采购人内部流程导致的付款顺延免责
- 是否存在供应商义务无边界扩张

当前代码状态：
- 已覆盖：
  - 采购人绝对免责
  - 供应商承担全部责任
  - 部分付款风险转嫁
- 仍待补：
  - 责任来源、过错划分、第三方原因混杂时的组合判断
  - 履约保证金、售后响应、违约责任不对等条款
  - 交货、安装、驻场、培训、维保义务边界

### 5. 验收与检测

人工重点会看：
- 验收标准是否模糊
- 第三方检测边界是否不清
- 抽检、复检、返工、费用承担是否明确
- 原材料检测和成品检测是否叠加过严

当前代码状态：
- 已覆盖：
  - 验收和第三方检测边界不清
- 仍待补：
  - 抽检触发条件
  - 复检机制
  - 检测失败后的救济与整改流程
  - 原材料、半成品、成品多层检测叠加的结构判断

### 6. 付款与违约

人工重点会看：
- 付款节点是否与外部抽检、终验、财政流程深度绑定
- 回款时间是否不确定
- 是否存在违约责任过重、单方解除过宽、扣款条件不清

当前代码状态：
- 已覆盖：
  - 付款与抽检终验联动
- 仍待补：
  - 付款节点整体链路解析
  - 违约金、扣款、单方解除、赔偿责任规则
  - 付款与验收、抽检、财政审批的多条件联动分析

### 7. 模板残留与义务外扩

人工重点会看：
- 是否出现与主标的不相关的服务内容
- 是否把其他项目模板中的义务、报告、保洁、垃圾清运等错贴进来
- 是否对供应商增加和项目无关的额外责任

当前代码状态：
- 已覆盖：
  - 模板残留和义务外扩的典型模式
- 仍待补：
  - 更多行业模板残留词典
  - 对“可能属于项目真实需要”的例外识别

## 三、当前最关键的缺口

如果目标是“先把各个审查查点和问题找出来”，当前最关键缺口不是表达，而是下面 5 类：

1. 评分表结构理解仍不够深
2. 技术参数边界判断仍偏弱
3. 商务、验收、付款仍未形成完整链路分析
4. 属地、本地服务团队、人员画像等资格与评分问题仍未系统覆盖
5. 例外与正当性机制还没真正建立

## 四、建议的实现顺序

### P0：先补查点覆盖率

- 资格条件剩余高频门槛
- 评分项更多类别
- 商务、验收、付款链路
- 技术参数高频边界词典

### P1：再补结构理解

- 评分项类别识别
- 同一评分项混合问题拆分
- 商务条款分组
- 技术条款组合判断

### P2：最后补边界判断

- 本地模型参与技术参数正当性判断
- 本地模型参与商务条款边界判断
- 本地模型参与复杂 finding 拆分/合并建议

## 五、产出要求

后续每次代码增强，至少应明确对应到本矩阵中的一项或多项查点：

- 补的是哪个一级查点
- 细化的是哪个二级查点
- 解决的是漏报、误报，还是拆分/合并问题
- 是否需要新 benchmark 样本

这样才能让“代码审查逼近人工审查”从零散优化变成可持续推进的工程。
