# 案例库设计

## 目标

建立一套可持续积累、可反向训练审查能力的案例库，使智能体能够从典型问题、投诉处理、监管认定和优质范例中持续学习。

## 案例库要解决的问题

- 让智能体知道什么是高频违规模式
- 让智能体知道什么是边界场景
- 让智能体学会把“风险感觉”转成“结构化判断”
- 让后续评测不再只依赖人工临时构造样例

## 案例来源

建议优先收集以下来源：

- 中国政府采购网典型案例、实务分析、投诉案例
- 财政部公开答复、政策解读
- 省级政府采购网公开的投诉处理、监督检查案例
- 实际采购文件中的匿名化问题条款
- 已完成人工审查的内部样例

## 案例分类

### 按问题类型分类

- 资格条件不合理
- 技术参数或服务要求定向化
- 品牌、型号、平台指定
- 评分标准不量化
- 与采购标的无关的加分项
- 同类业绩定义不清或范围过宽
- 属地、设点、本地备案限制
- 验收标准模糊
- 商务责任失衡
- 框架协议第二阶段规则不清

### 按风险等级分类

- 明显高风险案例
- 中风险边界案例
- 低风险优化案例
- 明显合规案例

### 按采购方式分类

- 公开招标
- 竞争性磋商
- 单一来源
- 框架协议
- 政府购买服务

## 单个案例的结构

每个案例建议采用固定结构。

```json
{
  "case_id": "CASE-0001",
  "title": "属地分公司门槛被认定为不合理限制竞争",
  "source_type": "典型案例",
  "source_url": "https://example.com",
  "jurisdiction": "山西省",
  "procurement_type": "框架协议",
  "issue_type": "geographic_restriction",
  "source_clause": "供应商须在采购人所在地行政区域内设有分公司，否则投标无效。",
  "regulatory_basis": [],
  "decision_or_analysis": "string",
  "expected_judgment": "likely_non_compliant",
  "expected_severity": 3,
  "rewrite_direction": "将属地设点改为本地化服务响应能力要求",
  "notes": "string"
}
```

## 最低字段要求

- `case_id`
- `title`
- `source_type`
- `source_url`
- `issue_type`
- `source_clause`
- `regulatory_basis`
- `expected_judgment`
- `expected_severity`
- `rewrite_direction`

## 案例处理流程

### 1. 收集

从权威网站抓取或人工录入案例。

### 2. 清洗

提炼出最关键的条款、争议点、认定理由和适用依据，删除无关背景信息。

### 3. 结构化

把案例转成统一字段，便于检索和评测。

### 4. 标注

为案例标注：
- 问题类型
- 严重度
- 是否需要人工复核
- 对应法条
- 建议改写方向

### 5. 入评测集

把高价值案例转化为 benchmark 测试题，用于测试智能体是否能稳定识别同类问题。

### 6. 反馈迭代

若智能体漏判、误判或引用错误，应把该失败重新写入案例库并更新规则。

## 案例库中的三类特殊样本

### 反例样本

即“看起来严格但其实可成立”的案例，用于避免过度误报。

### 优秀改写样本

保存“风险条款 -> 合规替代表述”的成对样本，用于训练改写能力。

### 失败样本

保存智能体过去审查错误的案例，用于持续修正判断偏差。

## 与评测系统的关系

案例库不是评测集的简单备份。

- 案例库：强调知识积累和监管经验沉淀
- 评测集：强调可重复测试和量化评分

一个案例进入案例库后，只有在字段足够明确时，才应进入评测集。

## 维护要求

- 每周至少新增一批典型案例候选。
- 每月整理一次“高频争议专题”，如属地限制、同类业绩、奖项加分等。
- 每次新增重要案例，都要补充一条对应的“审查启发规则”。
