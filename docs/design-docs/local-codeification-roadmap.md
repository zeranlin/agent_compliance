# 本地代码化总体思路

## 目标

把当前“政府采购合规审查智能体”的能力逐步沉淀为一套可离线运行的本地执行引擎，在无网络情况下仍可完成文档标准化、规则初筛、依据检索、finding 组装和结果输出。

## 核心架构

建议采用以下结构：

1. 文档标准化层
2. 规则命中层
3. 本地知识检索层
4. finding 组装层
5. 本地模型兜底层
6. 缓存与评测层

## 为什么不先上大模型

因为本地化的关键收益来自：
- 结构化输入稳定
- 高频风险由本地算法先拦截
- 依据由本地知识库直接提供
- 复审时复用缓存，避免重复判断

只有这几层先稳住，本地模型才会成为“少量复杂判断的补强器”，而不是“整篇文档的昂贵替代品”。

## 建议的模块边界

当前仓库中的代码目录已经基本符合这一路线：

- `parsers/`
- `rules/`
- `knowledge/`
- `pipelines/`
- `cache/`
- `evals/`

后续建议继续按这个边界补强，不另起一套并行实现。

## 本地模型的推荐职责

适合交给本地模型：
- 技术参数边界判断
- 商务条款合理性判断
- 聚合 finding 的摘要式说明
- 建议替代表述优化

不适合交给本地模型：
- 基础抽取
- 定位切分
- 高频规则扫描
- 法规和案例索引
- 全文自由审查

## 代码化优先级

### 优先级 A

- 标准化结果稳定化
- 规则覆盖扩充
- 本地知识检索增强
- finding 聚合与摘要输出

### 优先级 B

- findings 缓存
- 规则命中缓存
- 增量复审
- benchmark 回归

### 优先级 C

- 本地模型接入
- 模型路由策略
- 模型输出质量控制

## 当前推荐的实施顺序

1. 先做缓存与增量复审
2. 再做更正式的 finding 摘要式输出
3. 再扩规则和检索映射
4. 最后接本地模型

## 一句话结论

最优路线不是“把现有能力整体替换成本地大模型”，而是“先把确定性能力代码化，再让本地模型只补最难的部分”。
