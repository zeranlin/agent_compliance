# 政府采购需求审查智能体完整能力说明

## 目标定位

本项目当前不是一次性问答式的审稿助手，而是一个按 harness 方式组织的政府采购需求合规审查智能体工作框架。

它的目标不只是回答“某条款有没有风险”，而是形成一套：
- 可审查
- 可引用
- 可定位
- 可改写
- 可积累
- 可更新
- 可复盘
- 可续跑

的持续运行体系。

对外描述时，应同时承认两层含义：
- 一层是“体系化能力框架”，即项目被设计成具备审查、定位、引用、改写、一致性复审、缓存复用、案例学习、持续更新和评测改进能力的政府采购合规审查智能体。
- 一层是“当前已验证交付能力”，即已经能够对真实采购文件执行正文提取、条款切分、结构化审查、正式意见输出、结果落库、Git 留痕与复审复用的完整闭环。

## 一、审查对象覆盖能力

当前可覆盖的审查对象包括：
- 采购需求文本
- 招标文件、磋商文件、征集文件
- 资格条件
- 评分标准
- 技术要求和服务要求
- 合同条款
- 验收方案
- 框架协议项目第二阶段成交规则

这意味着当前能力并不局限于“看技术参数”或“看资格门槛”，而是能够覆盖采购文件中的主要高风险区域。

## 二、核心问题识别能力

当前重点识别的高频风险包括：
- 资格条件与履约能力无直接关系
- 属地限制、本地备案、本地分支机构门槛
- 所有制、股权结构、企业性质、投资者属性限制
- 品牌、型号、平台、专有能力、专有技术路线定向
- 技术参数区间过窄、特征叠加导致仅匹配少数产品
- 同类业绩范围过宽、数量过高、行业或区域定向
- 奖项、荣誉、信用等级、认证资质不当加分
- 资格条件或实质性要求被重复转化为评分优势
- 主观评分过重、分档不清、量化不足
- 人员画像限制，如性别、年龄、身高、外貌、特定履历
- 验收标准模糊、不可验证、缺少逐项验收机制
- 付款前提不合理、责任边界失衡、绝对免责
- 合同权利义务明显向单方倾斜
- 框架协议项目中第二阶段成交规则不清晰
- 与项目无关的模板残留或错贴条款

当前还具备边界判断意识，会区分：
- 合法的性能需求
- 可疑的限制性要求
- 明显排他性的表述
- 需要人工采购或法务复核的事项

## 三、审查判断标准能力

当前审查流程重点优化以下目标：
- 公平性
- 非歧视性
- 与实际履约能力的相关性
- 要求可量化、可审查
- 推理过程可追溯
- 在法律无法确定时保持克制并升级复核

这意味着当前能力并非机械地“只要严格就判风险”，而是尽量保留采购人的合理业务需求，只处理超出实际需要、排他性强或不可核验的部分。

## 四、文档读取、标准化与切分能力

当前能力不只是“阅读原文并给结论”，而是已经具备将采购文件转化为稳定中间表示的能力方向。

当前已验证或已明确设计的能力包括：
- 从 `docx`、`pdf` 或稳定文本副本中提取正文
- 对章节、条款、表格项、评分项、商务条款做切分
- 保留原始编号和章节路径
- 保留稳定文本副本，便于后续复核和复用
- 为后续实现预留 `file_hash`、`normalized_text`、`section_map`、`clause_map`、`page_map`、`line_map` 等标准化字段

这意味着后续复审默认不应每次都从零自由阅读整份文件，而是优先复用标准化输入。

## 五、精确定位与追溯能力

当前具备“主定位 + 辅助定位”的风险定位能力。

主定位字段包括：
- `document_name`
- `page_hint`
- `section_path`
- `clause_id`
- `table_or_item_label`
- `source_text`

辅助定位字段包括：
- `text_line_start`
- `text_line_end`

当前可以将问题尽量定位到：
- 哪一章
- 哪一条
- 哪个评分项
- 哪一行表格
- 哪段原文
- 哪个稳定文本行号

该能力当前适用于：
- Word
- PDF
- 表格类采购文件
- 评分表
- 合同模板

## 六、结构化输出能力

当前可以输出两类核心交付。

第一类是面向正式审查和人工复核的输出：
- 正式审查意见
- 正式审查意见表
- 业务方/采购人改稿用审查表

第二类是面向复用和下游处理的输出：
- 结构化 `findings.json`

正式输出中可以稳定提供以下字段：
- 位置
- 页码提示
- 条款编号
- 辅助行号
- 原文摘录
- 风险类型
- 风险等级
- 合规判断
- 法律/政策依据
- 适用逻辑
- 修改建议
- 建议替代表述
- 是否需要人工复核

结构化 finding 当前至少支持：
- 文档名
- 审查范围
- 风险摘要
- finding 编号
- 章节定位
- 问题类型
- 风险等级
- 严重度
- 置信度
- 合规判断
- 风险解释
- 竞争或履约影响
- 法律/政策依据
- 改写建议
- 人工复核标记和原因

## 七、风险分类、严重度和置信度建模能力

当前已具备标准化问题类型体系，包括但不限于：
- `brand_or_model_designation`
- `narrow_technical_parameter`
- `excessive_supplier_qualification`
- `geographic_restriction`
- `irrelevant_certification_or_award`
- `duplicative_scoring_advantage`
- `unclear_acceptance_standard`
- `one_sided_commercial_term`
- `ambiguous_requirement`
- `other`

当前还能对问题给出：
- `risk_level`: `high / medium / low / none`
- `severity_score`: `0-3`
- `confidence`: `high / medium / low`
- `compliance_judgment`: `likely_non_compliant / potentially_problematic / likely_compliant / needs_human_review`

这使输出更适合比较、复审、评测和版本迭代。

## 八、法规依据使用能力

当前具备法规依据分层使用能力，而不是仅依赖临时拼接的“原则性判断”。

依据分层包括：
- 法律
- 财政部规章和管理办法
- 规范性文件、公开答复、政策解读
- 地方财政和省级平台规则
- 典型案例和投诉处理口径

当前会按问题类型优先匹配不同法源：
- 资格条件问题优先看政府采购法及实施条例、需求管理办法、招标投标管理办法
- 技术要求问题优先看需求管理办法、框架协议规则、数据安全相关法规
- 评分标准问题优先看需求管理办法、评审规则、典型投诉案例
- 合同和验收问题优先看民法典、政府采购法及实施条例、需求管理办法

当前还具备克制引用能力：
- 无法稳定支持的规则不强行引用
- 依赖地方口径、最新监管文件、行业许可或兼容性正当性的条款，优先升级人工复核

## 九、本地知识库与引用资料能力

当前不是每次审查都从外部重新查找资料，而是基于仓库中的本地知识库工作。

当前已有：
- 法规依据本地引用库
- 案例口径本地引用库
- 引用资料索引

并且引用资料已具备统一元数据：
- `reference_id`
- `reference_type`
- `source_org`
- `source_url`
- `status`
- `review_topics`
- `related_rule_ids`
- `related_case_ids`
- `last_verified`

这意味着当前已具备：
- 按主题找法规和案例
- 按问题类型找对应依据
- 在输出中复用较稳定的引用提示
- 对来源进行追踪和核验

## 十、当前已沉淀的知识主题能力

当前本地知识库已覆盖这些高频审查主题：
- 政府采购需求管理办法
- 采购需求编制常见问题
- 主观评审客观化
- 综合评分法边界
- 属地限制与非歧视性审查
- 奖项荣誉与信用等级评分问题
- 资格条件与类似业绩设置
- 同类项目业绩案例解析
- 履约验收规范要点
- 合同付款前提与责任边界
- 框架协议采购工作问题

这意味着当前能力已经具有明确的政府采购专业知识面，而不是完全依赖通用语言能力。

## 十一、规则映射与知识组织能力

当前已具备把“问题类型、规则、案例、引用资料”打通的能力方向。

当前支持或已明确设计的映射包括：
- 从问题类型反查规则
- 从规则反查引用资料
- 从案例反查支撑依据

当前已有结构化资产包括：
- 法规依据库样表
- 典型案例库样表

字段体系已覆盖：
- `rule_id`
- `case_id`
- `reference_id`
- `source_url`
- `last_verified`

这使后续自动引用、自动更新和自动评测具备了结构基础。

## 十二、改写与整改建议能力

当前不只是指出风险，还具备改写能力。

能够处理的典型改写方向包括：
- 将属地门槛改为本地化服务响应能力要求
- 将品牌锚点改成功能和性能指标
- 将主观评分改为分项分档和量化指标
- 将模糊验收改为逐项核验清单
- 将单方失衡责任条款改为更平衡的风险分担机制

改写目标包括：
- 保留采购人的合法业务需求
- 不把需求改得过空
- 不引入新的不确定性条款

## 十三、人工升级复核能力

当前具备明确的升级复核机制。遇到以下情形时，应设置人工复核：
- 依赖地方性规则或最新监管文件
- 可能存在安全、保密、兼容性或法定资质正当性
- 文档片段不完整
- 条款语义过于模糊，无法形成稳定结论

这意味着当前系统不会为了显得“判断明确”而把所有边界问题都强行说死。

## 十四、规则命中与增量复审能力

当前系统已明确采用以下四层能力方向：
- 文档标准化层
- 规则命中层
- 大模型判断层
- 结果固化层

当前已具备或已明确设计的能力包括：
- 对高频问题先用规则、模式和关键词初筛
- 将规则命中结果作为后续判断的前置输入
- 仅在需要语义判断、边界判断、改写时调用大模型
- 将结果以结构化 findings 固化下来

这使当前系统具备：
- 更稳定的复审结果
- 更少的重复推理
- 更好的可解释性
- 更强的增量更新能力

## 十五、一致性复审与缓存复用能力

当前一致性不依赖“记住上次怎么说”，而依赖：
- 固定输入
- 固定规则
- 固定引用
- 固定模板
- findings 缓存
- 增量复审

当前已具备三层缓存思路：
- 文档解析缓存
- 规则命中缓存
- finding 缓存

典型复用逻辑包括：
- 只有输出格式变化时，直接重排已有 findings
- 只有引用资料变化时，只重算受影响 finding
- 只有文档局部变化时，只重审受影响条款

这使当前系统具备稳定复审和增量复审能力。

## 十六、案例学习能力

当前具备用案例反向训练审查能力的设计基础。

案例来源包括：
- 中国政府采购网典型案例和实务分析
- 财政部公开答复和政策解读
- 省级政府采购网投诉处理和监督检查案例
- 实际采购文件中的匿名化问题条款
- 已完成人工审查的内部样例

案例库覆盖的样本包括：
- 高风险案例
- 中风险边界案例
- 低风险优化案例
- 明显合规反例
- 优秀改写样本
- 失败样本

高价值案例还可进一步转成 benchmark 测试题，作为能力回归输入。

## 十七、失败复盘与持续学习能力

当前具备把历史错误转化为未来能力的机制。

可沉淀并转化的内容包括：
- 漏判样本
- 误判样本
- 引用依据错误样本
- 改写不当样本
- 需要人工升级但未升级的样本
- 过度升级的样本

这些内容会进一步转化为：
- 案例库条目
- benchmark
- 规则修订
- 引用映射修订
- 审查流程修订
- prompt 或规格更新

## 十八、评测能力

当前已具备较完整的评测结构。

已有：
- 审查评分规则
- 初始 benchmark 样例集
- 能力缺口评测报告模板

当前评测维度包括：
- 问题识别准确性
- 分类质量
- 推理可追溯性
- 改写建议可用性
- 升级复核克制性

当前失败标签包括：
- `false_negative`
- `false_positive`
- `severity_too_high`
- `severity_too_low`
- `unsupported_legal_basis`
- `rewrite_too_vague`
- `rewrite_changes_business_need`
- `insufficient_traceability`
- `missed_human_escalation`
- `over_escalated`

这意味着当前能力体系已经支持“被测量、被比较、被持续改进”。

## 十九、持续更新能力

当前已具备完整的持续更新机制设计。

更新对象覆盖：
- 法律法规和部门规章
- 财政部门政策解读和公开答复
- 国家级和地方级典型案例
- 内部审查失败样本和人工修正记录

更新节奏已定义为：
- 每周抓取新增案例和政策解读，识别新争议点
- 每月复核法规有效性、更新规则映射和 benchmark
- 每季度复盘误报漏报、调整问题分类和提示词、输出能力变化报告

单次更新流程包括：
- 抓取
- 识别
- 结构化
- 入库
- 生成派生产物
- 反馈到审查系统

## 二十、自动化准备能力

当前虽然尚未完全自动化，但已经具备自动化方案设计能力。

已拆分的自动化任务包括：
- 法规与政策扫描
- 案例候选收集
- 能力回归检查

自动化目标产物包括：
- `monthly-rule-update`
- `new-case-candidates`
- `eval-gap-report`

当前还明确了人工审核门槛，自动化结果不能直接进入正式知识库，仍需检查：
- 来源是否权威
- 摘要是否准确
- 是否误读结论
- 是否适合泛化成规则
- 是否属于地方特殊口径

这意味着当前系统已经具备向半自动和自动化演进的明确路径。

## 二十一、模板化产物生成能力

当前不仅能自由生成文本，还具备模板化产物意识。

当前模板包括：
- 正式审查意见模板
- 业务方/采购人修改用正式审查意见表模板
- 月度规则更新摘要模板
- 新增案例候选模板
- 能力缺口评测报告模板

这使当前能够生成：
- 正式复审报告
- 面向业务改稿的意见表
- 月度规则更新摘要
- 新案例候选清单
- 能力缺口评测报告

## 二十二、工程化与仓库协作能力

当前按仓库协作方式工作，而不是把结果停留在聊天记录里。

已形成的工程化能力包括：
- 每轮工作尽量沉淀到仓库
- 完成完整阶段结果后自动提交 Git
- 默认自动推送远端
- 使用结论式提交信息
- 将活动计划、设计理由、产品规格、评测、生成产物分层归档
- 让后续智能体能从仓库状态接力，而不是依赖聊天上下文

这意味着当前系统具备：
- 中断后续跑
- 多轮协作可追溯
- 产物长期可检查
- 复盘时可直接查看历史执行痕迹

## 二十三、当前已验证的交付闭环

从现有仓库状态与真实执行过程来看，当前已经验证过这些能力闭环：
- 读取真实采购文件
- 抽取正文
- 进行章节和条款定位
- 输出正式审查意见
- 输出结构化 findings
- 生成稳定文本副本
- 将结果落到 `docs/generated/`
- 自动提交并推送 Git
- 复审时修正定位字段并保持结论一致

这说明当前能力不仅停留在设计层，也已有实际可执行链路。

## 二十四、当前能力边界

当前尚未完全自动化的部分包括：
- 自动抓取官方站点并定期生成真实更新结果
- 自动将新增案例直接写入正式案例库
- 自动跑全量评测并输出真实差异报告
- 对所有地方口径和最新监管变化做实时无误判断

当前仍会对以下问题保持克制并优先升级人工复核：
- 高度依赖地方规则
- 行业许可、保密、安全、兼容性具有特殊必要性
- 文档片段不完整
- 依据不稳定或未核实有效性

## 结论性概括

一句话概括，当前的我不是单纯“会看采购条款”的智能体，而是一个具备审查、标准化、定位、引用、改写、规则命中、缓存复用、增量复审、案例学习、持续更新、自动化准备、评测回归和工程化留痕能力的政府采购需求合规审查体系。
