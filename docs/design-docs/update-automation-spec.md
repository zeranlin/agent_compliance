# 更新自动化方案

## 目标

把“持续更新机制”从原则设计推进到可执行方案，明确后续自动化任务应该做什么、产出什么、如何进入仓库。

## 自动化目标产物

建议每轮自动化运行至少产出以下文件之一：

- `docs/generated/monthly-rule-update-YYYY-MM.md`
- `docs/generated/new-case-candidates-YYYY-MM-DD.md`
- `docs/generated/eval-gap-report-YYYY-MM.md`

## 自动化任务拆分

### 自动化任务 A：法规与政策扫描

输入来源：
- 财政部官网
- 中国政府采购网政策栏目
- 省财政厅官网

主要动作：
- 抓取新增政策、规章、答复、解读
- 判断是否属于新增、替代、失效、口径变化
- 生成法规变更摘要

建议输出字段：
- 标题
- 来源链接
- 发布时间
- 文件类型
- 影响主题
- 是否需要更新法规依据库

### 自动化任务 B：案例候选收集

输入来源：
- 中国政府采购网典型案例栏目
- 省级政府采购网投诉处理栏目
- 政府采购实务分析栏目

主要动作：
- 抓取新增案例
- 提取条款、争议点、认定逻辑、适用依据
- 生成案例候选列表

建议输出字段：
- 案例标题
- 来源链接
- 问题类型候选
- 原始条款
- 认定摘要
- 建议进入案例库/评测集/仅参考

### 自动化任务 C：能力回归检查

输入来源：
- 新增案例候选
- 现有 benchmark
- 最近一阶段人工审查结果

主要动作：
- 用现有规则对新案例做一次模拟判断
- 比较结果与人工预期
- 输出误报、漏报、依据缺失点

建议输出字段：
- case_id
- 预期判断
- 当前判断
- 差异类型
- 需要补充的规则
- 是否需要新增 benchmark

## 仓库存放约定

- 自动化原始候选放在 `docs/generated/`
- 人工确认后的正式规则进入法规依据库
- 人工确认后的正式案例进入案例库
- 可重复测试的样本进入 `docs/evals/`

## 人工审核门槛

自动化结果不能直接视为最终知识，至少要经过以下人工确认：

- 来源是否权威
- 摘要是否准确
- 是否误读案例结论
- 是否适合泛化成审查规则
- 是否属于地方特殊口径

## 首版实施建议

第一阶段不必直接开发完整程序，可以先采用“半自动”方式：

1. 定时抓取官方站点新增页面列表
2. 由智能体先做摘要和分类
3. 由人工确认后写入库
4. 再把确认后的内容转入评测和规则体系

## 成功标准

当自动化方案投入使用后，至少应做到：

- 新案例能在一周内进入候选池
- 新规则变化能在一个月内进入法规依据库
- 误报漏报能在季度内反映到 benchmark 中
