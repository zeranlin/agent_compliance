# 当前架构缺口清单与补强优先级

## 1. 文档目标

本文档用于回答一个比“还要补哪些规则”更高一层的问题：

> 当前代码审查主链路已经补到了文件级策略、采购品目、章节级主题分析、模型辅助、仲裁和差异学习，但如果继续往人工式审查逼近，最关键的缺口还剩哪些，它们的优先级是什么？

本文档不再讨论单条规则或单个样本问题，而是聚焦“架构层还缺哪几层”。

## 2. 当前主链路回顾

当前代码审查主链路已经升级为：

`文档解析与标准化 -> 文件级策略 -> 采购品目分类 -> 规则扫描 -> 主题分析器 -> 模型辅助补点 -> finding 仲裁 -> 证据选择 -> 输出 -> 差异学习`

当前已经具备：
- 文档标准化和稳定文本复用
- 文件级主风险画像与复核路线
- 采购品目分类与官方品目编码映射
- 按品目差异化路由规则和主题分析器
- 章节级主问题归并
- 本地模型辅助边界判断
- 规则候选、benchmark gate、difference learning

因此，下一阶段的关键问题已经不是“再补一些规则”，而是要补齐仍然缺失的架构层。

## 3. 当前最关键的 5 个架构缺口

### 3.1 法规条文级语义层

#### 现状

当前已经有：
- 本地法规依据库
- 元数据规范
- 权威目录核验层
- 离线原文快照和标准化文本入口

但当前法规层更多还停留在：
- 引用层
- 摘要层
- 元数据核验层

还没有真正进入：
- 条文级适用层
- 问题类型到法条的结构化映射层
- 条文位阶和适用边界的推理层

#### 为什么这是严重缺口

现在系统可以“引用法规”，但还不够会：
- 精确说明某条结论更直接对应哪一条款
- 区分法律、行政法规、财政部规章、规范性文件的适用位阶
- 处理“同一问题可由多个依据支撑，但主依据不同”的情况
- 给“需论证”问题配置更自然的法理边界

这意味着系统越来越会“看到问题”，但还不一定越来越会“严谨说明为什么”。

#### 建议新增模块

- `legal_clause_index`
- `legal_authority_reasoner`
- `issue_type_authority_map`

#### 优先级

`P0`

---

### 3.2 规则治理层

#### 现状

当前规则、主题分析器、模型建议、规则候选都在持续增长，但还缺一层明确的规则治理能力。

现在已经有：
- `ALL_RULES`
- 规则候选
- benchmark gate
- 规则管理页

但还缺：
- 规则位阶和优先级
- 规则适用品目范围
- 规则是否为基础硬规则、场景规则或实验候选规则
- 不同采购品目下的规则启停和降权机制

#### 为什么这是严重缺口

没有规则治理层，系统会出现两个问题：
- 规则越多，越难维护
- 品目层接入后，规则无法优雅表达“这条规则在哪类采购里应该更强/更弱”

这会导致：
- 误报难以下降
- 同类规则冲突
- 某些规则在错误场景下上浮过重

#### 建议新增模块

- `rule_registry`
- `rule_priority_profile`
- `catalog_sensitive_rule_router`

#### 优先级

`P0`

---

### 3.3 置信度校准层

#### 现状

当前结果里已经有：
- `risk_level`
- `confidence`
- `needs_human_review`

但这仍然更像结果字段，而不是一个真正的置信度系统。

还缺：
- 规则命中与模型命中如何合成最终置信
- 哪些场景应自动降为“需论证”或“待复核”
- 如何区分：
  - 高置信硬风险
  - 中置信结构性问题
  - 低置信边界问题

#### 为什么这是严重缺口

随着：
- 规则更多
- 品目层更复杂
- LLM 补点更多

如果没有置信度校准层，最终结果会越来越难解释：
- 为什么这是高风险
- 为什么这是需论证
- 为什么这条模型新增没有直接上浮

#### 建议新增模块

- `confidence_calibrator`
- `finding_confidence_profile`

#### 优先级

`P1`

---

### 3.4 benchmark 评估报告层

#### 现状

现在已经有：
- benchmark 样本
- benchmark gate
- issue type 覆盖统计

但仍缺：
- 更完整的能力评估报告层
- 每轮增强后的收益和副作用分析
- 按品目场景的能力短板说明

#### 为什么这是严重缺口

如果没有评估报告层，后续增强会越来越依赖感觉：
- 这轮到底提升了什么
- 哪个 analyzer 更有效
- 哪类误报下降了
- 哪类漏报仍然严重

这会直接影响后续迭代效率。

#### 建议新增模块

- `benchmark_regression_reporter`
- `catalog_scene_eval_report`

#### 优先级

`P1`

---

### 3.5 品目知识画像层

#### 现状

当前品目层已经有：
- 全量官方品目骨架
- 审查领域映射
- 主品目/次品目/混合采购识别

但“品目知识层”仍然只是第一版，还偏弱：
- 只解决“这是什么”
- 还没有完整解决“这种品目通常怎么审”

#### 为什么这是严重缺口

如果没有品目知识画像层，品目层就只能做：
- 分类
- 路由

却不能更深地回答：
- 这个品目下常见合理要求是什么
- 常见高风险要求是什么
- 常见错位内容是什么
- 哪些 analyzer 在这个品目下最关键

而这正是人工审查分寸感的重要来源。

#### 建议新增模块/数据

- `catalog_knowledge_profile`
- `catalog_reasonable_requirements`
- `catalog_high_risk_patterns`
- `catalog_preferred_analyzers`

#### 优先级

`P1`

## 4. 缺口优先级总表

| 缺口层 | 价值 | 风险 | 优先级 |
|---|---|---|---|
| 法规条文级语义层 | 直接提升结论严谨性和引用质量 | 不补会导致“会看问题但不会精确说明” | P0 |
| 规则治理层 | 直接提升规则体系可维护性和品目适配性 | 不补会导致规则越多越乱 | P0 |
| 置信度校准层 | 提升高/中/需论证之间的稳定边界 | 不补会导致结果解释性下降 | P1 |
| benchmark 评估报告层 | 提升持续优化可解释性 | 不补会导致增强效果难量化 | P1 |
| 品目知识画像层 | 提升场景分寸感和 analyzer 适配度 | 不补会导致品目层停留在分类层 | P1 |

## 5. 推荐补强顺序

### 第一阶段：先把“规则和依据”做稳

优先做：
- 法规条文级语义层
- 规则治理层

目标：
- 让系统不仅会审，还会更严谨地说明为什么
- 让品目驱动后的规则体系保持可维护

### 第二阶段：再把“结果解释和评测”做稳

优先做：
- 置信度校准层
- benchmark 评估报告层

目标：
- 让输出更可解释
- 让每轮增强都能量化收益

### 第三阶段：把“品目层”从分类升级为知识层

优先做：
- 品目知识画像层

目标：
- 让品目层真正成为人工审查分寸感的底座

## 6. 对现有主链路的影响

这 5 层补上后，主链路将进一步升级为：

`文档解析 -> 文件级策略 -> 品目分类 -> 品目知识画像 -> 规则治理路由 -> 规则扫描 -> 主题分析器 -> 模型辅助 -> 置信度校准 -> finding 仲裁 -> 证据选择 -> 输出 -> benchmark 评估报告 -> 差异学习`

与当前相比，最关键的变化是：
- 规则不再只是“能命中”，而是“有位阶、有场景、有治理”
- 法规不再只是“能引用”，而是“能按条文级说明”
- 结果不再只是“有高低风险”，而是“有更稳定的置信度边界”
- benchmark 不再只是 gate，而是“能报告哪类能力真的提升了”
- 品目不再只是“分类”，而是“带合理要求画像和高风险画像”

## 7. 一句话结论

当前代码审查主链路已经补上了文件级策略层、品目层、主题分析层、仲裁层和差异学习层。  
如果继续逼近人工审查，下一批最关键的缺口已经不是“再补一些规则”，而是：

- 法规条文级语义层
- 规则治理层
- 置信度校准层
- benchmark 评估报告层
- 品目知识画像层

其中最优先的两层是：
- 法规条文级语义层
- 规则治理层

因为这两层会直接决定：后续系统是“越来越强”，还是“越来越复杂但越来越难管”。
