# review-check 动态审查进度方案

## 目标

在 `review-check` 页面中，当用户选择“启用大模型（混合审查）”后，文件上传与审查过程会明显变长。当前页面在等待阶段缺少清晰反馈，容易让采购人误以为：

- 页面卡住
- 大模型不可用
- 审查没有继续执行

本方案的目标是：

- 用采购人可理解的方式展示真实审查进度
- 让“代码主链 + 大模型关键节点介入”的混合审查过程可感知
- 不暴露过多调试细节，但保留可展开的技术视图
- 为后续“先出基础结果，再补增强结果”预留状态结构

## 设计原则

- 展示真实阶段，不做假百分比
- 优先用采购人语言，不把页面做成调试面板
- 先实现稳妥的轮询，再考虑 SSE
- 后端状态字段应与主链路节点对应，避免前后端各说各话
- 混合审查过程要体现“大模型介入了哪些关键环节”，但不必在默认视图里堆技术术语

## 适用范围

本方案适用于：

- `review-check`
- 启用 `启用大模型（混合审查）` 的审查流程

本方案暂不优先覆盖：

- `review-next` 的验证型调试视图
- CLI 流式进度输出
- 多任务队列管理页

## 页面展示方案

### 一、默认采购人视角

建议在 `review-check` 上传区下方、结果区上方，增加一个轻量 `审查进度` 面板。

默认展示 5 个阶段：

1. `文档解析中`
   - 正在提取正文、章节和表格内容
2. `基础风险扫描中`
   - 正在识别资格、评分、技术、商务/验收风险
3. `智能增强分析中`
   - 正在补充边界问题、章节主问题和法规解释
4. `结果收束中`
   - 正在去重、合并主问题、整理证据和建议
5. `审查完成`
   - 可查看问题清单和导出结果

每个阶段显示状态：

- `等待中`
- `进行中`
- `已完成`
- `已跳过`
- `失败`

建议样式：

- 当前阶段高亮
- 已完成阶段使用低饱和绿色或蓝色勾选
- 失败阶段使用红色提示
- 不使用跳动百分比条，避免误导

### 二、可展开的技术视图

默认折叠，点击“查看详细进度”后展开：

- 品目识别
- 规则扫描
- 评分语义分析
- 混合边界分析
- 商务链路分析
- 全文辅助扫描
- 章节级总结
- 法规适用逻辑解释
- 仲裁归并
- 证据选择

这层更适合内部验证和排障，但不应默认打扰采购人。

### 三、结果联动

若后续实现“基础结果先展示，增强结果后补全”，页面行为建议如下：

- 基础链完成后，页面先显示：
  - 初步主问题
  - 初步摘要
  - 原文定位
- 同时继续显示：
  - `智能增强分析中`
- 增强完成后：
  - 替换为最终主问题和说明
  - 更新导出结果

## 后端状态字段设计

建议新增统一的任务状态对象：

```json
{
  "job_id": "review-20260319-001",
  "status": "running",
  "mode": "hybrid",
  "stage_profile": {
    "stage_key": "pre_release_requirement_review",
    "stage_name": "采购需求形成与发布前审查"
  },
  "progress": {
    "current_step": "llm_chapter_summary",
    "current_label": "智能增强分析中",
    "steps": []
  },
  "partial_result_available": false,
  "last_message": "正在补充章节主问题",
  "started_at": "2026-03-19T10:00:00+08:00",
  "updated_at": "2026-03-19T10:00:08+08:00"
}
```

### 顶层字段

- `job_id`
  - 本次审查任务唯一 ID
- `status`
  - `queued / running / partial / completed / failed / cancelled`
- `mode`
  - `rule_only / hybrid`
- `stage_profile`
  - 当前采购流程阶段
- `progress`
  - 详细步骤状态
- `partial_result_available`
  - 是否已有可展示基础结果
- `last_message`
  - 当前阶段说明
- `started_at`
- `updated_at`

### `progress.steps`

建议结构：

```json
[
  {
    "step_key": "parse",
    "label": "文档解析中",
    "status": "completed",
    "started_at": "...",
    "ended_at": "...",
    "message": "已提取正文、章节和表格内容"
  },
  {
    "step_key": "rule_scan",
    "label": "基础风险扫描中",
    "status": "completed",
    "started_at": "...",
    "ended_at": "...",
    "message": "已完成规则扫描和主题分析"
  },
  {
    "step_key": "llm_document_audit",
    "label": "全文辅助扫描",
    "status": "running",
    "started_at": "...",
    "ended_at": null,
    "message": "正在补充边界问题和混合采购线索"
  }
]
```

## 建议的步骤拆分

建议主链统一拆成以下步骤：

1. `parse`
   - 文档解析
2. `strategy`
   - 文件级策略与品目识别
3. `rule_scan`
   - 规则扫描
4. `analyzers`
   - 主题分析器
5. `llm_document_audit`
   - 全文辅助扫描
6. `llm_chapter_summary`
   - 章节级总结
7. `llm_legal_reasoning`
   - 法规适用逻辑解释
8. `arbiter`
   - 仲裁归并
9. `evidence`
   - 证据选择
10. `done`
   - 审查完成

说明：

- 若未启用大模型：
  - 第 5-7 步直接标为 `skipped`
- 若某个模型节点输入为空：
  - 也可以标为 `skipped`

## 页面文案建议

默认采购人视角文案应避免技术感过强。

建议：

- `文档解析中`
  - 正在提取正文、章节和表格内容
- `基础风险扫描中`
  - 正在识别资格、评分、技术、商务/验收风险
- `智能增强分析中`
  - 正在补充边界问题、章节主问题和法规解释
- `结果收束中`
  - 正在去重、整理证据并生成审查意见
- `审查完成`
  - 当前结果已可查看和导出

技术层展开后再显示：

- 品目识别
- 全文辅助扫描
- 章节级总结
- 法规适用逻辑解释

## 轮询还是 SSE

### 第一阶段建议：轮询

优先建议采用：

- 前端发起审查任务
- 后端返回 `job_id`
- 前端每 `1s` 轮询 `/api/review-status?job_id=...`

原因：

- 实现简单
- 容易与当前同步接口兼容
- 更适合当前单机本地服务
- 容错更稳，页面刷新恢复也更容易

### 第二阶段再考虑：SSE

当以下条件成立后，再考虑升级：

- 混合审查流程已经稳定
- 任务状态对象字段固定
- 页面需要更实时的节点更新

SSE 的优势：

- 更实时
- 减少轮询请求

但当前不应优先，因为：

- 前后端状态机还在演进
- 单机本地服务下轮询成本可接受

## 接口建议

### 1. 创建任务

`POST /api/review-start`

返回：

```json
{
  "job_id": "review-20260319-001",
  "status": "queued"
}
```

### 2. 查询状态

`GET /api/review-status?job_id=...`

返回上述任务状态对象。

### 3. 获取结果

`GET /api/review-result?job_id=...`

当 `status=completed` 或 `partial` 时可获取当前结果。

## 推荐实现顺序

### P0

- 任务 ID
- 状态对象
- 轮询接口
- `review-check` 进度面板
- 大模型开启时展示真实阶段

### P1

- 支持基础结果先展示、增强结果后补全
- 技术视图展开层
- 失败与超时提示

### P2

- SSE 推送
- 后台任务恢复
- 多任务历史查看

## 一句话结论

`review-check` 最适合的动态进度方案是：

- 默认用采购人可读的 5 阶段进度面板
- 可展开看技术节点
- 后端统一用任务状态对象表达步骤
- 第一阶段先用轮询，不急着上 SSE
- 后续再做“基础结果先展示，增强结果后补全”
