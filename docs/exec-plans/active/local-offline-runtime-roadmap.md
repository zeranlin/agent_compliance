# 本地离线执行引擎推进计划

## 目标

在现有 harness 和离线骨架基础上，把政府采购合规审查能力逐步代码化为可本地运行、可断网执行、可持续增强的执行引擎，并尽量减少大模型调用。

## 当前状态

已完成：
- 本地 CLI 骨架和基础目录结构
- `docx/txt/pdf` 文本抽取
- 条款切分、`section_path`、`table_or_item_label`
- `page_map` / `page_hint`
- 高频规则初筛
- 本地法规与案例引用资料检索
- finding 聚合
- 更接近正式审查意见的 `problem_title` 和改写建议

当前仍主要缺少：
- 更细的规则覆盖面和规则版本管理
- 更稳定的复杂表格和跨页定位
- 正式报告级别的摘要式摘录和依据归并
- 权威法规原文快照层和标准化法规文本层
- findings 缓存和增量复审
- 本地模型兜底推理层
- 离线 benchmark 回归和误报漏报闭环

## 总体原则

1. 先做确定性能力，再接本地模型。
2. 先做规则和检索，再做生成式润色。
3. 先保证离线稳定可跑，再提升“像正式审查意见”的表达质量。
4. 本地模型只处理边界判断、归纳摘要和改写增强，不处理整篇自由审查。
5. 离线运行不只依赖摘要型引用资料，应逐步补齐本地权威法规原文和标准化法规文本。

## 分阶段路线

### 阶段一：标准化和定位层做稳

目标：
- 让文档切分和定位成为稳定中间层，供规则、检索和模型共同复用。

重点工作：
- 继续增强复杂表格、评分表、合同清单的定位能力
- 统一 `section_map`、`clause_map`、`page_map` 输出结构
- 为后续缓存设计稳定的输入指纹

完成标志：
- 同一文件重复运行时，条款切分和主定位字段基本稳定
- 大部分 finding 能提供可读的 `section_path`、`page_hint`、`table_or_item_label`

### 阶段二：规则与知识检索层扩面

目标：
- 用本地算法覆盖大部分高频合规风险，显著压缩模型调用量。

重点工作：
- 扩充资格、评分、技术、商务、验收、框架协议规则
- 细化 `rule_id -> issue_type -> reference_id` 映射
- 增加本地引用资料排序和摘要抽取
- 为高频法规建立本地原文快照和标准化文本索引

完成标志：
- 高频风险主题具备可重复命中的本地规则
- 规则命中后能自动挂上较稳定的法规/案例依据
- 断网时可以从本地标准化法规文本中完成基础检索和引用

### 阶段三：finding 组装与复审层增强

目标：
- 让离线 findings 更接近正式审查意见，支持复审复用。

重点工作：
- 对相邻命中做更稳的聚合
- 生成问题标题、代表性摘录、命中条数、条款范围
- 增加 findings 缓存、规则命中缓存、标准化缓存
- 支持局部重算和增量复审

完成标志：
- 同一文件复审时不再重复全量组装
- 离线 findings 已可直接作为正式审查前的半成品

### 阶段四：本地模型兜底层接入

目标：
- 只在确定性能力不足时调用本地模型，补足边界判断和改写质量。

建议模型职责：
- 技术参数正当性边界判断
- 商务条款是否属于合理内控边界
- 聚合后条款的摘要式风险说明
- 更自然的建议替代表述

不建议交给模型的任务：
- 整篇文件从零审查
- 基础定位和规则初筛
- 本地引用资料查找

完成标志：
- 大模型调用频次显著低于逐条自由审查模式
- 模型只处理少数高价值、低确定性的 finding

### 阶段五：评测与持续更新层闭环

目标：
- 让离线引擎具备可持续增强能力，而不是只靠人工经验堆规则。

重点工作：
- 把失败样本转成 benchmark
- 记录误报、漏报、依据不足、改写不当
- 输出阶段性 `eval-gap-report`
- 用 benchmark 驱动规则和模型策略迭代

完成标志：
- 每轮增强都能通过固定样例回归
- 新增规则或模型策略能看到真实收益

## 是否必须接本地大模型

不是必须立刻接。

推荐顺序：
1. 先把标准化、规则、检索、组装、缓存做稳
2. 再接本地模型作为边界判断器

原因：
- 当前最值钱的降本手段是“减少不必要的模型调用”
- 不是“把现有流程全部换成本地模型”

## 当前最优先下一步

建议按以下顺序继续推进：

1. 做 findings 缓存和增量复审
2. 把聚合后的原文摘录和依据整理成更正式的摘要式输出
3. 扩充规则覆盖面、规则映射和本地法规原文快照
4. 再接本地模型处理边界判断和改写增强

## 风险提醒

- 如果过早把整篇审查交给本地模型，结果会不稳定，且很难复审。
- 如果没有先做缓存和规则分层，后续即使接了本地模型，成本也不会低。
- 如果定位层不稳定，正式审查意见会很难被业务和采购人员直接复核。
